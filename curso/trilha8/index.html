Model System

4 providers (firstParty/Bedrock/Vertex/Foundry), 5 selection layers, aliases com [1m] suffix, effort levels, fast mode e model migrations.

1
📋 Four-Provider Registry & Selection Chain
ALL_MODEL_CONFIGS, 5-layer priority, allowlist gate
O que e: Cada modelo declarado em configs.ts com ID per-provider. Selection: /model command > --model flag > ANTHROPIC_MODEL env > settings.json > subscription default. availableModels allowlist gate roda antes de tudo.
Por que aprender: O registry unico com 4 providers e a chain de 5 camadas sao a espinha dorsal de toda interacao com a API.
Conceitos-chave: ModelConfig Record, @[MODEL LAUNCH] 7-file checklist, Bedrock cross-region profile auto-discovery, modelOverrides layering.
2
🏷️ Aliases, [1m] Suffix & opusplan
sonnet/opus/haiku/best, 1M context gate, plan mode upgrade
O que e: Aliases resolvem short names para model strings do provider ativo. [1m] suffix e runtime flag stripped na API boundary. opusplan alterna Sonnet/Opus baseado no permission mode. haiku upgrada para Sonnet em plan mode.
Por que aprender: O sistema de aliases com [1m] suffix demonstra como abstrair complexidade de providers para o usuario.
Conceitos-chave: parseUserSpecifiedModel(), normalizeModelStringForAPI(), checkOpus1mAccess(), isOpus1mMergeEnabled() getSubscriptionType null guard.
3
💪 Effort Levels & Fast Mode
auto/low/medium/high/max, Opus 4.6 only, cooldown
O que e: Effort: thinking budget hint (auto/low/medium/high/max). max e Opus 4.6 only, session-scoped para non-ants. Fast mode: lower-latency path, first-party only, extra-usage billing required. Env var sempre vence UI setting.
Por que aprender: Effort e fast mode sao knobs ortogonais que controlam trade-offs de custo, latencia e qualidade.
Conceitos-chave: resolveAppliedEffort() env > state > default, toPersistableEffort(), isFastModeEnabled() multi-layer check, cooldown timer.
4
🛡️ Allowlist: Three-Tier Matching
Family wildcard, version prefix, exact ID, narrowing rule
O que e: availableModels com 3 tiers: family wildcard ("opus"), version prefix ("opus-4-5"), exact ID. Narrowing rule: ["opus", "opus-4-5"] NAO permite todos Opus - o entry especifico restringe o wildcard.
Por que aprender: A narrowing rule e non-obvious e previne admins de acidentalmente permitir modelos nao desejados.
Conceitos-chave: isModelAllowed() 3-tier matching, familyHasSpecificEntries() narrowing, empty allowlist blocks ALL, modelMatchesVersionPrefix().
5
🔀 Subagent Model Inheritance
4-level precedence, alias-matches-parent, Bedrock region prefix
O que e: Subagent model: CLAUDE_CODE_SUBAGENT_MODEL env > tool-specified > agent config > inherit. Bare alias matching parent tier herda model string exato. Bedrock region prefix (eu./us.) herdado automaticamente.
Por que aprender: Model inheritance com region prefix e essencial para deployments multi-regiao com IAM permissions scoped.
Conceitos-chave: aliasMatchesParentTier(), applyParentRegionPrefix(), inherit default, [1m]/best/opusplan fall through.
6
🔄 Model Migrations & Validation
6 idempotent migrations, live API probe, deprecation warnings
O que e: 6 migracoes idempotentes rodam em cada startup (so tocam userSettings). validateModel() envia API call real com max_tokens:1 para probar modelo customizado. Deprecation table com retirement dates per-provider.
Por que aprender: Migracoes automaticas de model settings sao um pattern essencial para evolucao de produto sem quebrar usuarios.
Conceitos-chave: migrateLegacyOpusToCurrent, migrateSonnet45ToSonnet46, migrateOpusToOpus1m, DISABLE_LEGACY_MODEL_REMAP env.

Sandbox Security

Confinamento OS-level via Apple Seatbelt/bubblewrap, 3 modos (disabled/regular/auto-allow), network/filesystem control e secure storage.

1
🏗️ Sandbox Architecture & Three Modes
Seatbelt macOS, bubblewrap Linux, disabled/regular/auto-allow
O que e: Adapter bridges @anthropic-ai/sandbox-runtime com settings do Claude Code. macOS usa Apple Seatbelt (zero install), Linux usa bubblewrap + seccomp + socat. 3 modos via 2 booleans (enabled + autoAllowBashIfSandboxed).
Por que aprender: Entender os 3 modos e o thin adapter pattern e essencial para configurar seguranca em producao.
Conceitos-chave: BaseSandboxManager, SandboxRuntimeConfig, memoized dependency checks, seccomp warning-only, policy lock detection.
2
🌐 Network & Filesystem Control
Domain allow/deny, always-denied paths, bare git repo defense
O que e: Domains de WebFetch permissions + sandbox.network settings. Settings files e .claude/skills SEMPRE denied write. Bare git repo sentinel files scrubbed apos cada comando (core.fsmonitor escape vector).
Por que aprender: A defesa contra git sandbox escape (issue #29316) e um caso real de seguranca com two-prong defense.
Conceitos-chave: allowManagedDomainsOnly enterprise lockdown, resolvePathPatternForSandbox vs resolveSandboxFilesystemPath, worktree write access.
3
🔑 macOS Keychain Storage
Hex encoding, stdin buffer limit, generation counter
O que e: Credentials em hex via security CLI. stdin preferido (4032B limit) sobre argv. 30s TTL cache com stale-while-error. Generation counter previne async read stale de sobrescrever write recente. Startup prefetch paralelo.
Por que aprender: O cache com generation tracking e stale-while-error e um pattern avancado de resiliencia para storage lento.
Conceitos-chave: keychainCacheState generation, readInFlight dedup, KEYCHAIN_CACHE_TTL_MS=30000, startKeychainPrefetch() parallel.
4
🔄 Fallback Storage & Login Loop Fix
createFallbackStorage combinator, stale primary deletion
O que e: createFallbackStorage wraps keychain + plaintext. Key invariant: read() prefere primary. Se plaintext tem token fresh e keychain tem stale, stale vence -> login loop. Fix: delete stale primary apos fallback write success (issue #30337).
Por que aprender: O login loop bug e seu fix demonstram a importancia de invariantes em storage com fallback.
Conceitos-chave: update() migration logic, primaryDataBefore check, 0o600 permissions, plainText warning on every write.
5
🚨 Violation Reporting & Live Config
SandboxViolationStore, stderr annotation, settingsChangeDetector
O que e: Violacoes armazenadas em SandboxViolationStore. stderr anotado com explicacao legivel. XML sandbox_violations stripped para display. Config atualiza live via settingsChangeDetector subscription sem restart.
Por que aprender: Violation reporting com live config refresh demonstra operabilidade madura de sandbox.
Conceitos-chave: annotateStderrWithSandboxFailures(), removeSandboxViolationTags(), BaseSandboxManager.updateConfig() sync, enabledPlatforms NVIDIA override.
6
🏢 Enterprise Policy Lock & Excluded Commands
flagSettings/policySettings priority, /sandbox exclude
O que e: Quando flagSettings ou policySettings setam sandbox keys, localSettings nao pode override. /sandbox exclude adiciona patterns a excludedCommands. allowUnsandboxedCommands controla se excluded commands podem rodar fora do sandbox.
Por que aprender: Policy lock e command exclusion sao os mecanismos enterprise para customizar sandbox sem comprometer seguranca.
Conceitos-chave: areSandboxSettingsLockedByPolicy(), addToExcludedCommands(), allowUnsandboxedCommands flag, getSandboxUnavailableReason().

Message Processing

Pipeline de 4 estagios: submit/route, input classification, message construction e API normalization com merge de turns e tool result pairing.

1
📨 handlePromptSubmit: Entry Point
Queue vs direct path, paste expansion, immediate commands
O que e: Dois paths: queue processor (pre-validated) e direct input (com validacao). expandPastedTextRefs substitui placeholders. queryGuard.isActive enfileira novos inputs. Immediate commands (/config, /doctor) executam durante query ativa.
Por que aprender: O entry point com dual path e query guard e a base de toda interacao de usuario.
Conceitos-chave: QueuedCommand shape, commandQueue + useSyncExternalStore, queryGuard reserves BEFORE processUserInput, immediate command detection.
2
🔀 Input Classification & Hooks
bash/slash/text/ultraplan branching, UserPromptSubmit hooks
O que e: 4 branches: bash mode, slash command (com bridge safety check), ultraplan keyword (hidden), text prompt. Images resize em paralelo. UserPromptSubmit hooks podem block, prevent continuation ou inject context. Hook output capped a 10K chars.
Por que aprender: A classificacao de input com hooks interceptores e o mecanismo central de extensibilidade do pipeline.
Conceitos-chave: effectiveSkipSlash for bridge safety, ultraplan pre-expansion detection, applyTruncation() 10K cap, blockingError/preventContinuation.
3
📦 Message Construction & Factory
createUserMessage, isMeta, synthetic messages
O que e: createUserMessage e a factory canonica. isMeta: true = model-visible mas hidden do transcript. Synthetic messages (INTERRUPT_MESSAGE, CANCEL_MESSAGE) influenciam modelo. SYNTHETIC_TOOL_RESULT_PLACEHOLDER e guard de training data.
Por que aprender: Entender isMeta e synthetic messages e crucial para debug de comportamento do modelo.
Conceitos-chave: uuid stable identity, origin provenance, permissionMode safety snapshot, imagePasteIds ordering, image metadata isMeta messages.
4
🔄 normalizeMessagesForAPI
6-pass transformation, turn merging, tool result pairing
O que e: 6 passes: reorder attachments, strip virtual messages, build error-to-strip map, filter system/attachment, merge consecutive same-role, ensure tool result pairing. deriveUUID para split estavel. Short message IDs (6-char base36).
Por que aprender: normalizeMessagesForAPI e o gatekeeper final que garante payloads validos para a Anthropic API.
Conceitos-chave: Consecutive merge for alternating turns, SYNTHETIC_TOOL_RESULT_PLACEHOLDER for orphaned tool_use, deriveShortMessageId() deterministic.
5
📚 Message Taxonomy & Memory Hint
6 message types, withMemoryCorrectionHint pattern
O que e: 6 tipos: UserMessage, AssistantMessage, AttachmentMessage, SystemMessage, ProgressMessage, TombstoneMessage. Memory correction hint appended a rejection/cancel messages para direcionar modelo a salvar preferencias do usuario.
Por que aprender: A taxonomia completa explica por que o pipeline precisa de tantos passes de transformacao.
Conceitos-chave: AttachmentMessage (reordered, never sent to API), SystemMessage subtypes, ProgressMessage ephemeral, TombstoneMessage deletion marker.
6
⏱️ Query Profiling Checkpoints
Wall-clock timestamps, 3 latency hotspots
O que e: queryCheckpoint(label) registra timestamps em pontos chave. 3 maiores fontes de latencia: image processing (resize + base64), attachment loading (IDE + git diff), hook execution (external process). Images usam Promise.all; attachments sao sequenciais.
Por que aprender: Os checkpoints sao o mapa de performance do pipeline e revelam oportunidades de otimizacao.
Conceitos-chave: query_image_processing_start/end, query_attachment_loading_start/end, query_hooks_start/end, pasted images parallel vs attachments sequential.

Task System

7 task types (bash, agent, remote, teammate, workflow, monitor, dream) com lifecycle de 5 estados, notification XML, output em file mode e O_NOFOLLOW security.

1
🏗️ Task Architecture & 5-State Lifecycle
TaskStateBase, 7 types, notified atomic latch
O que e: 7 tipos de task compartilham TaskStateBase com id (prefix encodes type), status (pending/running/completed/failed/killed), notified flag atomic latch. Task IDs: b=bash, a=agent, r=remote, t=teammate, w=workflow, m=monitor, d=dream.
Por que aprender: O lifecycle unificado com notified latch e a base de todo tracking de trabalho concorrente.
Conceitos-chave: TaskType union, compare-and-set notified pattern, eviction after terminal+notified, 1s pollTasks() interval.
2
💻 LocalShellTask: Stall Watchdog & Orphan Cleanup
OS fd output, 7 prompt patterns, agent orphan kill
O que e: Stdout/stderr vao direto para arquivo via OS fd (zero JS buffering). Stall watchdog verifica crescimento a cada 5s, detecta 7 prompt patterns apos 45s. killShellTasksForAgent() mata processos orfaos quando agente pai morre.
Por que aprender: O stall watchdog com prompt detection e orphan cleanup previnem problemas reais de producao.
Conceitos-chave: PROMPT_PATTERNS 7 regexes, stall notification sem status tag, foreground->background transition via Ctrl+B, agentId tracking.
3
🤖 Agent & Remote Tasks
Progress tracking, retain/evict, remote completion checkers
O que e: LocalAgentTask: latestInputTokens (cumulative) + cumulativeOutputTokens (per-turn). Retain/evict gate com 30s grace period. RemoteAgentTask: 5 subtypes com pluggable completion checkers. Metadata persistence para --resume.
Por que aprender: A distincao entre cumulative vs per-turn token counting e essencial para metricas corretas.
Conceitos-chave: ProgressTracker, PANEL_GRACE_MS=30s, registerCompletionChecker(), writeRemoteAgentMetadata() sidecar, review extraction 2 producers.
4
👥 Teammate & Dream Tasks
50-msg cap (36GB fix), idle lifecycle, lock rewind
O que e: InProcessTeammateTask: 50-message UI cap (292 agents = 36.8GB RSS sem cap). isIdle boolean independente de status. DreamTask: 2 fases (starting/updating), UI-only sem model notification, rollbackConsolidationLock() no kill.
Por que aprender: O 50-message cap e uma necessidade de producao real, nao limitacao de UI. Lock rewind previne deadlock permanente.
Conceitos-chave: TEAMMATE_MESSAGES_UI_CAP=50, appendCappedMessage(), isIdle vs status distinction, priorMtime rollback, no model notification.
5
📢 Notification System & Output Management
XML envelope, next vs later priority, O_NOFOLLOW security
O que e: XML notifications com task_id, tool_use_id, output_file, status, summary. Priority 'next' (monitor) vs 'later' (shell). Output: DiskTaskOutput (5GB cap) + TaskOutput (8MB memory, spill to disk). O_NOFOLLOW previne symlink attack.
Por que aprender: O_NOFOLLOW e a defesa contra path-traversal em ambientes sandboxed. O stall notification sem status tag e deliberado.
Conceitos-chave: task_notification XML, stall notification no-status-tag design, DiskTaskOutput 5GB cap, agent transcript symlinks, formatTaskOutput() 32K default.
6
📋 TaskCreate/Update vs Background Tasks
Two separate "task" systems, verification nudge, auto-ownership
O que e: Background task system (Task.ts) tracks execucao concorrente. Task list (TaskCreate/Update) e checklist de project management em YAML/JSON. Completamente separados. TaskUpdate: auto-ownership para teammates, TaskCompleted hooks gate, verification nudge.
Por que aprender: Distinguir os dois sistemas "task" previne confusao arquitetural fundamental.
Conceitos-chave: registerTask() resume path (carry forward UI state), updateTaskState() reference-equality optimization, pill label aggregation.

REPL Screen

O arquivo de 5000 linhas que E a sessao Claude Code: turn lifecycle, QueryGuard, dialog priority queue, session resume e auto-restore on interrupt.

1
🔄 Turn Lifecycle: onSubmit -> onQuery -> onQueryImpl
QueryGuard generation counter, messagesRef pattern
O que e: 3 funcoes encadeadas: onSubmit (entry point, idle-return gate), onQuery (QueryGuard com generation counter), onQueryImpl (system prompt, API call, streaming). messagesRef.current evita recrear onSubmit ~30x por turn (~9 REPL scopes leaked antes do fix).
Por que aprender: O QueryGuard com generation counter e o ref pattern sao solucoes para problemas reais de performance e memory leaks.
Conceitos-chave: queryGuard.tryStart() generation, end(thisGeneration) stale-finally guard, messagesRef sync pattern, 25+ dep useCallback.
2
💡 Loading State & Dialog Priority Queue
3 sources of isLoading, getFocusedInputDialog() pure function
O que e: 3 fontes de loading: isQueryActive (local), isExternalLoading (remote/SSH), hasRunningTeammates. Dialog priority: pure function retorna um unico vencedor de lista deterministica. 1.5s debounce supprime dialogs enquanto usuario digita.
Por que aprender: O dialog system como pure function elimina boolean soup e garante comportamento previsivel.
Conceitos-chave: getFocusedInputDialog() string union, PROMPT_SUPPRESSION_MS=1500, ScrollKeybindingHandler before CancelRequestHandler ordering.
3
💬 Messages Array & Deferred Rendering
Ref+state dual, ephemeral progress replacement, useDeferredValue
O que e: messages como ref (sync) + state (render). Ephemeral progress (Sleep/Bash ticks) replacem in-place (evita 13K+ entries). Compact boundary substitui array. useDeferredValue mantém input box responsivo durante streaming.
Por que aprender: O pattern dual ref+state e o deferred rendering sao tecnicas avancadas de React performance.
Conceitos-chave: rawSetMessages wrapper, ephemeral progress replacement by toolUseId, compact boundary handling, deferred bypass when streaming visible.
4
🔄 Session Resume & toolJSX Overlay
15-step resume, local JSX vs tool updates priority
O que e: Resume executa 15 steps: deserialize messages, fire hooks, restore costs, switch session, worktree management etc. clearSessionMetadata antes de restore previne stale name. toolJSX overlay: local JSX commands nao podem ser overwritten por tool updates.
Por que aprender: O resume flow de 15 steps e a razao por tras de cada ordering decision revelam complexidade real.
Conceitos-chave: clearSessionMetadata before restoreSessionMetadata, localJSXCommandRef preservation, immediate vs non-immediate JSX placement.
5
↩️ Auto-Restore on Interrupt
Esc rewind, runs outside generation guard, 5 conditions
O que e: Quando Esc cancela e nao houve resposta significativa, REPL rewind a conversa e restaura input. 5 conditions: user-cancel, !queryGuard.isActive, inputValue vazio, queue vazio, nao viewing teammate. Roda FORA do generation guard por design.
Por que aprender: Auto-restore fora do generation guard e uma decisao arquitetural nao-obvia que resolve um race condition real.
Conceitos-chave: forceEnd() bumps generation, end(thisGeneration) returns false, messagesAfterAreOnlySynthetic check, removeLastFromHistory().
6
🎨 Render Tree & Performance Isolation
FullscreenLayout, AnimatedTerminalTitle null-render, unseen divider
O que e: JSX tree: AlternateScreen > KeybindingSetup > FullscreenLayout com scrollable, bottom, overlay e modal slots. AnimatedTerminalTitle retorna null (pure side-effect, isolada). Unseen divider muda apenas 2x por scroll session.
Por que aprender: Isolamento de rendering com componentes null-return e o pattern de unseen divider demonstram React performance avancado.
Conceitos-chave: AnimatedTerminalTitle 960ms tick isolated, Prompt vs Transcript two screens, VirtualMessageList for 250MB saving, dividerIndex changes 2x only.
