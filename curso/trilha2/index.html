The Tool System

A interface Tool, o pipeline de registro e filtragem, orquestracao de concorrencia, permissoes e o fluxo completo de execucao.

1
🔧 The Tool Interface (Tool.ts)
Protocol contract, inputSchema, call, permissions
O que e: A interface Tool<Input, Output, P> e um protocol contract TypeScript que toda ferramenta deve satisfazer. Define name, aliases, inputSchema (Zod), call(), checkPermissions(), isConcurrencySafe() e isReadOnly().
Por que aprender: Cada capacidade do Claude Code e uma Tool. Entender a interface e fundamental para criar, customizar ou debugar qualquer ferramenta.
Conceitos-chave: Structural typing (protocol, not class), Zod schema validation, ToolResult with contextModifier, renderToolUseMessage/renderToolResultMessage UI hooks, interruptBehavior.
2
🏭 buildTool() Factory & Fail-Closed Defaults
Defaults conservadores, BuiltTool type magic
O que e: buildTool() merge um ToolDef parcial com defaults seguros (fail-closed): isConcurrencySafe=false, isReadOnly=false, isDestructive=false. O tipo BuiltTool<D> preserva tipos literais da definicao.
Por que aprender: Entender os defaults fail-closed explica por que ferramentas custom rodam serialmente por padrao e como opt-in para concorrencia.
Conceitos-chave: TOOL_DEFAULTS, fail-closed philosophy, TypeScript spread at type level, checkPermissions default (allow, defer to general system).
3
📦 Three-Tier Registration Pipeline
getAllBaseTools, getTools, assembleToolPool
O que e: Pipeline de tres camadas: getAllBaseTools() (catalogo exaustivo com feature flags), getTools() (filtragem por modo e deny rules), assembleToolPool() (built-ins + MCP, sorted para cache stability).
Por que aprender: O pipeline de registro determina quais ferramentas o modelo ve. Entende-lo e essencial para debugging de ferramentas ausentes ou MCP tools nao aparecendo.
Conceitos-chave: Feature flags gate tools at load time, deny rules filter before model sees them, MCP tools sorted as separate alphabetical suffix for prompt-cache stability.
4
🔄 Orchestration & Concurrency
partitionToolCalls, concurrent vs serial batches
O que e: O orchestrator usa partitionToolCalls() para decidir quais tools rodam em paralelo e quais serialmente. isConcurrencySafe(input) e avaliado per-call, nao per-tool-type. Concurrency ceiling default: 10.
Por que aprender: Concorrencia data-driven e o que permite Bash(ls) rodar paralelo enquanto Bash(rm -rf) roda serial. Entender o particionamento previne bugs de state mutation.
Conceitos-chave: partitionToolCalls algorithm, contextModifier dance (serial immediate, concurrent queued), all() async-generator combinator, CLAUDE_CODE_MAX_TOOL_USE_CONCURRENCY.
5
⚡ StreamingToolExecutor
Real-time execution, sibling abort, in-order emission
O que e: StreamingToolExecutor inicia tools enquanto blocos ainda stream da API. Tool lifecycle: queued -> executing -> completed -> yielded. Apenas erros de Bash abortam siblings. Resultados emitidos na ordem do modelo.
Por que aprender: E o executor usado em producao. Entender sibling abort e in-order emission e critico para debugging de ferramentas paralelas.
Conceitos-chave: canExecuteTool() concurrency guard, siblingAbortController (Bash-only cascade), progress messages emitted immediately, getRemainingResults() in insertion order.
6
🛡️ Permission Context & Execution Pipeline
checkPermissionsAndCallTool, DeepImmutable, Zod validation
O que e: O pipeline completo: Zod validation -> semantic validation -> speculative classifier -> backfillObservableInput (clone) -> PreToolUse hooks -> canUseTool() -> tool.call() -> PostToolUse hooks -> result serialization. ToolPermissionContext e DeepImmutable.
Por que aprender: O pipeline de execucao e o caminho critico de seguranca. Cada etapa tem razao de existir, incluindo _simulatedSedEdit stripping como defense-in-depth.
Conceitos-chave: DeepImmutable permission context, three input copies (API-bound, observable clone, hook-updated), filterToolsByDenyRules, Stream multiplexing progress+results.

The Bash Tool

O gateway do sistema operacional: 7 camadas de processamento, 23 security validators, shell snapshot, sandboxing e output handling.

1
💻 Architecture & Command Pipeline
7 layers antes do spawn, buildExecCommand()
O que e: O Bash Tool processa comandos por 7 camadas antes do subprocess spawn: input validation, permission checking, command building (6 estagios incluindo shell quoting e pipe rearrangement), e output streaming.
Por que aprender: Todo comando shell passa por este pipeline. Entender cada camada e essencial para diagnosticar falhas de permissao e comportamentos inesperados.
Conceitos-chave: buildExecCommand() 6 stages, eval quoting for alias availability, pipe rearrangement fix, extglob disabling, stdin redirect decision.
2
🛡️ The 23 Security Validators
bashCommandIsSafe(), allow/passthrough/ask chain
O que e: Cadeia de 23 validators em bashCommandIsSafe(), cada um retornando allow, passthrough ou ask. Cobrem: shell metacharacters, dangerous variables, obfuscated flags, unicode whitespace, brace expansion, ZSH-specific commands e mais.
Por que aprender: Os validators formam defense-in-depth contra injection em niveis de character encoding, shell syntax e Zsh-specific. Sao a fronteira de seguranca real.
Conceitos-chave: Five pre-computed command views (originalCommand, baseCommand, unquotedContent, fullyUnquotedContent, unquotedKeepQuoteChars), safe heredoc early-allow, 50-subcommand cap.
3
📸 Shell Snapshot System
Aliases, functions, PATH capturados uma vez por sessao
O que e: Cada sessao captura o ambiente shell interativo em um snapshot file: aliases, functions, shell options, PATH e ripgrep shim embarcado. Graceful degradation: se falhar, usa login shell com flag -l.
Por que aprender: O snapshot elimina a necessidade de login shell por comando, reduzindo latencia significativamente. Entender o fallback ajuda a diagnosticar aliases ausentes.
Conceitos-chave: ~/.claude/shell-snapshots/, TOCTOU awareness (access() before each command), filtered completion functions, graceful degradation to -l flag.
4
🔒 Permission Plumbing & Sandboxing
bashToolHasPermission(), SandboxManager, rule matching
O que e: checkPermissions() chama bashToolHasPermission() com 6 checks independentes. SandboxManager controla filesystem read/write e network. Tres modos de rule matching: exact, prefix (cmd:*) e wildcard.
Por que aprender: Sandbox exclusions sao convenience features, NAO controles de seguranca. O dialogo de permissao e a fronteira real. Entender isso previne falsa sensacao de seguranca.
Conceitos-chave: stripSafeWrappers() two-phase (env vars + wrapper commands), auto-generated prefix rules (2-word extraction), dangerouslyDisableSandbox policy check.
5
⏳ Background Execution & Output Handling
3 paths de background, size limits, image detection
O que e: Tres caminhos de background execution: run_in_background (model-initiated), Ctrl+B (user), auto-background (15s timeout). Output limitado a 30KB inline (150KB hard cap). Deteccao automatica de imagens base64.
Por que aprender: Entender os tres paths de background e os limites de output previne confusao quando comandos longos sao cortados ou movidos para background.
Conceitos-chave: backgroundTaskId, semantic exit code interpretation (grep 1 = no matches), Claude Code Hints protocol (zero-token side channel), BASH_SILENT_COMMANDS (mv,cp,rm show "Done").
6
🎯 UI Classification & Sed Special Case
Collapse behavior, pipeline classification, sed-as-edit
O que e: Comandos classificados para UI collapse: search/read/list commands colapsaveis. Pipelines so colapsam se TODAS as partes forem search/read. Sed -i renderiza como file-edit com diff-style permission dialog.
Por que aprender: A classificacao UI impacta a experiencia do usuario. Entender por que certos comandos colapsam e outros nao ajuda a customizar o comportamento.
Conceitos-chave: BASH_SEARCH_COMMANDS, BASH_READ_COMMANDS, BASH_LIST_COMMANDS, semantic-neutral commands (echo, printf), BASH_SILENT_COMMANDS, sed FileEdit display name.

File Tools (Read, Write, Edit)

Tres ferramentas de manipulacao de arquivo com um invariante compartilhado: toda escrita requer leitura previa. Dedup, quote normalization e staleness checks.

1
📖 Read Tool Deep-Dive
Pagination, token limits, image/PDF/notebook support
O que e: Read suporta text, images (PNG/JPG/GIF/WebP com resize via sharp), PDFs (pages parameter, max 20 pages) e Jupyter notebooks. Token limit de 25K com two-stage gate (rough estimate + exact count). Dedup retorna stub de ~100 bytes quando arquivo nao mudou (~18% hit rate).
Por que aprender: Read e a unica tool read-only e concurrency-safe. Seu sistema de dedup economiza milhares de tokens por sessao. Entender os limites previne erros de "file too large".
Conceitos-chave: readFileState mtime check, MaxFileReadTokenExceededError (throw not truncate), 256KB size gate, blocked device paths (/dev/zero, /dev/random), macOS thin space retry.
2
✏️ Write Tool Deep-Dive
Read-before-write gate, atomic write, LF policy
O que e: Write cria ou sobrescreve arquivos inteiros. Requer Read previa completa (nao parcial). Sequencia atomica: mkdir -> backup -> sync read + staleness check -> write. Sempre usa LF line endings. LSP notifications (didChange + didSave) apos escrita.
Por que aprender: O gate read-before-write e a protecao principal contra sobrescrita acidental. Entender os tres error codes (2: no read, 2: partial read, 3: file modified) acelera debugging.
Conceitos-chave: readFileState check, errorCode 2 vs 3, Windows mtime content-comparison fallback, CLAUDE.md telemetry, skill discovery on write.
3
✂️ Edit Tool & Quote Normalization
Exact string replacement, curly quotes, desanitization
O que e: Edit faz substring match exato com replace. Se >1 match, recusa (error 9) a menos que replace_all:true. findActualString() normaliza curly quotes para match. preserveQuoteStyle() mantem tipografia original. Desanitization table reverte tokens sanitizados pela API.
Por que aprender: Edit e a ferramenta mais eficiente para mudancas cirurgicas. Quote normalization e o motivo pelo qual funciona mesmo com curly quotes em arquivos.
Conceitos-chave: normalizeQuotes() (U+2018/2019/201C/201D), contraction detection (/\\p{L}/u), old_string="" creates new file, trailing whitespace stripping (except .md/.mdx).
4
🔄 The Read-Before-Write Contract
4 fases, double staleness check, race window
O que e: Contrato em 4 fases: Read armazena em readFileState -> validateInput() rejeita se nao lido/parcial/stale -> call() roda staleness check sincrono (fecha race window) -> Write/Edit atualiza readFileState. Dois checks de staleness (validateInput + call) fecham a janela de corrida.
Por que aprender: Gotcha mais comum: linter/formatter modifica arquivo entre Read e Write, disparando erro "file has been modified since read". Re-read resolve.
Conceitos-chave: Double staleness check (pre-permission + atomic in call), isPartialView flag, Windows content-comparison fallback, FILE_UNEXPECTEDLY_MODIFIED_ERROR.
5
📊 Write vs Edit: When to Use Each
Token cost comparison, use cases
O que e: Edit envia ~20 chars para uma mudanca cirurgica; Write envia o arquivo inteiro. Write para: novo arquivo, rewrite completo, mudancas extensas. Edit para: tudo mais. A economia de tokens e dramatica em arquivos grandes.
Por que aprender: Escolher a ferramenta errada desperica tokens e aumenta latencia. A regra simples: se o arquivo existe e a mudanca e pequena, use Edit.
Conceitos-chave: Token cost: Edit ~20 chars vs Write entire file, new file creation, structural rewrites, unique old_string requirement.
6
🔏 Limits Precedence & Dedup System
Env var > GrowthBook > default, mtime-based dedup
O que e: Token/byte limits seguem chain de 3 niveis: env var > GrowthBook flag > hardcoded default (25K tokens / 256KB). Memoized na primeira chamada. Dedup de Read: se mtime igual, retorna stub. Apenas Read-originated entries participam do dedup.
Por que aprender: Saber como overridar limites e essencial para projetos com arquivos grandes. O dedup economiza ~18% de re-reads desnecessarios.
Conceitos-chave: CLAUDE_CODE_FILE_READ_MAX_OUTPUT_TOKENS, tengu_amber_wren GrowthBook flag, file_unchanged stub type, Edit/Write entries excluded from dedup.

Search Tools (Glob & Grep)

Ambas delegam para ripgrep: Glob busca nomes de arquivo, Grep busca conteudo. Tres output modes, pagination, e resolucao de binario em tres camadas.

1
🔍 Glob Delegates to ripgrep
--files --glob, absolute path decomposition
O que e: Glob NAO usa JavaScript glob libraries. Delega para ripgrep com --files --glob. Patterns absolutos sao decompostos por extractGlobBaseDirectory() em baseDir + relativePattern. Ignora .gitignore por default (Claude precisa ver build artifacts).
Por que aprender: Saber que Glob e ripgrep explica sua performance em codebases gigantes e por que .gitignore e ignorado por padrao.
Conceitos-chave: extractGlobBaseDirectory(), CLAUDE_CODE_GLOB_NO_IGNORE, CLAUDE_CODE_GLOB_HIDDEN, --sort=modified, 100 file limit (globLimits).
2
📋 Grep's Three Output Modes
files_with_matches, content, count
O que e: Tres modos = tres invocacoes ripgrep distintas: files_with_matches (rg -l, paths only, sorted by mtime), content (matching lines com context -B/-A/-C), count (rg -c, match counts per file).
Por que aprender: Escolher o modo correto evita trabalho desnecessario. files_with_matches para "quais arquivos?", content para "quais linhas?", count para "quantas vezes?".
Conceitos-chave: mtime sort in files_with_matches (most recent first), test mode uses filename sort, line numbers only in content mode, context flags precedence (-C supersedes -B/-A).
3
📄 Pagination: head_limit & offset
Default 250, unlimited escape hatch, 20KB persist threshold
O que e: Ambas ferramentas suportam head_limit (default 250) e offset como tail -n +N | head -N. head_limit=0 e o escape hatch para ilimitado. Head-limiting acontece ANTES de path relativization (otimizacao: evita processar 10K linhas descartadas).
Por que aprender: O default de 250 mantem resultados sob o threshold de 20KB persist. Saber paginar e essencial para buscas amplas.
Conceitos-chave: applyHeadLimit() with appliedLimit hint, truncation annotation in tool result, head_limit=0 unlimited, 20KB persist threshold (~6-24K tokens/grep-heavy session).
4
⚙️ ripgrep Binary Resolution
system -> embedded -> builtin, memoized, security
O que e: Resolucao em tres modos memoizada: system (rg do PATH, usa nome "rg" nao path resolvido contra PATH hijacking), embedded (Bun, argv0='rg'), builtin (vendor/ripgrep/). macOS code-signing automatico para builtin.
Por que aprender: O detalhe de seguranca e sutil: usar "rg" em vez do path resolvido previne execucao de ./rg.exe malicioso. Entender os tres modos ajuda debugging de "ripgrep not found".
Conceitos-chave: getRipgrepConfig() memoized factory, USE_BUILTIN_RIPGREP env var, macOS codesign --sign - (ad-hoc), spawn vs execFile (argv0 support).
5
⚡ Performance Architecture
Timeouts, EAGAIN retry, buffer cap, path relativization
O que e: Timeouts platform-aware: 20s padrao, 60s WSL. EAGAIN retry com -j 1 (single thread) para uma unica chamada, nao persistido globalmente. Buffer cap 20MB. Path relativization economiza centenas de tokens por chamada. Line length cap: 500 chars.
Por que aprender: Performance tuning e essencial em monorepos grandes. O retry EAGAIN scoped evita regressao que o retry global causava.
Conceitos-chave: CLAUDE_CODE_GLOB_TIMEOUT_SECONDS, SIGTERM -> SIGKILL 5s escalation, ripGrepFileCount streaming counter, --max-columns 500, VCS directory exclusions.
6
🛡️ Pattern Safety & UNC Path Security
Leading-dash problem, NTLM leak prevention, glob parsing
O que e: Patterns comecando com dash usam -e flag para prevenir interpretacao como flag do ripgrep. UNC paths (\\, //) skipam stat() para prevenir NTLM credential leaks via SMB handshake. Glob patterns com braces tratados como padrao unico.
Por que aprender: A classe de vulnerabilidade leading-dash e comum em tools que fazem shell-out. A protecao UNC e critica em ambientes Windows/network.
Conceitos-chave: -e pattern flag, UNC NTLM credential leak prevention, brace pattern detection (*.{ts,tsx}), type parameter vs glob (--type more efficient), isConcurrencySafe()=true for both.

MCP Integration

Model Context Protocol: 8 transports, 7-layer config cascade, OAuth com PKCE, tool proxying, elicitation e deduplication baseada em conteudo.

1
🔌 Transport Types & Architecture
8 transports: stdio, sse, http, ws, ide, sdk, in-process
O que e: 8 tipos de transport: stdio (subprocess stdin/stdout), sse (HTTP SSE com OAuth), http (Streamable HTTP, MCP 2025-03-26), ws (WebSocket), sse-ide/ws-ide (IDE extensions), sdk (control-message bridge), in-process (Chrome MCP/Computer Use via queueMicrotask).
Por que aprender: Cada transport tem requisitos de setup distintos. Saber qual usar para cada cenario (local, remoto, IDE, plugin) e essencial para integracoes MCP.
Conceitos-chave: InProcessTransport (createLinkedTransportPair), stdio stderr piped/capped 64MB, sse GET skip 60s timeout (long-lived streams), ws protocols: ['mcp'].
2
📋 7-Layer Config Scope Cascade
enterprise > dynamic > claudeai > project > local > user > managed
O que e: Configs MCP mergem de 7 fontes com precedencia estrita: enterprise (managed-mcp.json, bloqueia tudo), dynamic (--mcp-config), claudeai (connector API), project (.mcp.json), local (~/.claude/projects/), user (~/.claude/settings.json), managed (plugins).
Por que aprender: Enterprise lock bloqueia completamente add/remove manual. Saber a cascade resolve conflitos de configuracao e entende por que um server nao aparece.
Conceitos-chave: Enterprise Lock ("exclusive control"), .mcp.json walks directory tree (child overrides parent), env var expansion ($VAR), allowedMcpServers/deniedMcpServers policy.
3
🔗 Connection Lifecycle & State Machine
Batched connection, timeout race, capability negotiation
O que e: Lifecycle em 8 etapas: config assembly -> batched connection (stdio:3, remote:20) -> transport construction -> client.connect() with 30s timeout race -> auth handling (401 -> needs-auth) -> capability negotiation -> tool/resource/prompt fetch -> live notifications.
Por que aprender: O timeout race e batched connection sao criticos para startup performance. Entender o estado needs-auth explica o pseudo-tool McpAuthTool.
Conceitos-chave: Promise.race([connectPromise, timeoutPromise]), MCP_TIMEOUT env var, exponential backoff reconnect (1s -> 30s cap, max 5), ToolListChanged/ResourceListChanged notifications.
4
🎭 Tool Proxying & MCPTool
Name normalization, description cap, result handling
O que e: MCPTool e um template clonado para cada tool de cada server. Nomes normalizados como mcp__server__tool. Descricoes hard-capped em 2048 chars. Resultados com imagens sao resized, blobs binarios persistidos em disco, resultados >100KB truncados.
Por que aprender: O cap de 2048 chars previne explosion de contexto de servers OpenAPI. Entender name normalization e essencial para permission rules.
Conceitos-chave: normalizeNameForMCP (non-alphanumeric -> underscore), buildMcpToolName(), image resize/downsample, binary blob persistence, 100KB truncation threshold.
5
🔐 OAuth Authentication & McpAuthTool
PKCE flow, model-triggered auth, Slack quirk, XAA
O que e: OAuth completo com PKCE: 401 -> needs-auth state -> McpAuthTool pseudo-tool injected -> model triggers auth -> browser opens -> callback fires -> tokens stored in keychain -> reconnect with real tools. Slack returns 200 com error body, normalizado para 400.
Por que aprender: OAuth model-triggered e unico do Claude Code. Entender o fluxo McpAuthTool -> real tools swap previne confusao quando servers requerem auth.
Conceitos-chave: McpAuthTool pseudo-tool, prefix-based replacement on AppState, Slack 200-error normalization, XAA cross-app access (SSO without browser), 15-min needs-auth cache.
6
📡 Elicitation & Deduplication
Form/URL modes, content-based dedup, CCR proxy unwrapping
O que e: Elicitation: servers pedem input estruturado mid-operation (form mode com JSON Schema ou URL mode). Dedup baseado em content signature (nao nome): stdio:["cmd","arg1"] ou url:https://... CCR proxy unwrapping preserva URL original para comparacao.
Por que aprender: Dedup content-based previne duplicatas quando o mesmo server aparece em multiplas fontes. Elicitation permite interacoes ricas com servers MCP.
Conceitos-chave: ElicitationRequestEvent with respond() callback, hooks can satisfy requests programmatically, manual wins over plugin, first plugin wins, enabled manual wins over claude.ai.
