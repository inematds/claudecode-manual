Bridge & Remote

Como o Claude Code conecta o REPL local ao claude.ai via bridge bidirecional, com duas arquiteturas (v1 env-based e v2 env-less) e transport layer assimetrico.

1
🌉 Bridge v1 vs v2 Architecture
Env-based polling vs env-less direct OAuth
O que e: Duas arquiteturas de bridge coexistem: v1 (register/poll/ack/heartbeat via Environments API) e v2 (POST /bridge direto, sem Environment). O GrowthBook flag tengu_bridge_repl_v2 controla qual caminho o REPL segue.
Por que aprender: Entender as duas versoes e essencial para diagnosticar problemas de conectividade e entender a evolucao da arquitetura.
Conceitos-chave: WorkSecret decode, initBridgeCore vs initEnvLessBridgeCore, session ID compatibility (cse_ vs session_), gate logic.
2
📡 Transport Layer: WebSocket, SSE & Hybrid
ReplBridgeTransport interface e backpressure
O que e: v1 usa HybridTransport (WebSocket reads + HTTP POSTs). v2 e assimetrico: SSE para leitura com sequence numbers para reconnect gapless, e CCRClient para escrita via /worker/events.
Por que aprender: O transport layer e a base de toda comunicacao remota e entender seus modos e crucial para debugging.
Conceitos-chave: SSETransport, CCRClient, SerialBatchEventUploader, epoch mismatch (409), FlushGate para prevenir interleaving.
3
🔒 Permission Bridge Protocol
control_request/response e synthetic messages
O que e: Prompts de permissao viajam pelo bridge via control_request/control_response. RemoteSessionManager coordena requests pendentes. createSyntheticAssistantMessage fabrica mensagens para tools remotos desconhecidos.
Por que aprender: O fluxo de permissoes e central a seguranca do sistema remoto e ao modelo de confianca.
Conceitos-chave: BridgePermissionResponse, createToolStub para MCP tools remotos, pendingPermissionRequests Map.
4
⚙️ /remote-control Command & Entitlements
BridgeToggle, 5 pre-flight checks, CCR mirror
O que e: O comando /remote-control executa 5 verificacoes antes de conectar: runtime gate, OAuth token, org policy, token freshness e min version. CCR Mirror Mode permite streaming outbound-only para sessoes read-only no claude.ai.
Por que aprender: Entender os gates de entitlement previne problemas de conexao e esclarece o modelo de seguranca.
Conceitos-chave: checkBridgePrerequisites, isCcrMirrorEnabled, outboundOnly mode, session title derivation (Haiku-generated).
5
☁️ CCR Integration & SDK Adapter
SessionsWebSocket, SDKMessage conversion
O que e: CCR e o ambiente de execucao server-side. SessionsWebSocket conecta ao subscribe endpoint. sdkMessageAdapter traduz mensagens SDK para o tipo interno Message do REPL, com regras especificas para ignorar user messages inbound.
Por que aprender: Entender a integracao CCR e fundamental para entender como sessoes remotas funcionam na pratica.
Conceitos-chave: SessionsWebSocket reconnect logic (5 attempts), convertSDKMessage table, user messages ignored to prevent duplication.
6
🖥️ Standalone Bridge Server Mode
runBridgeLoop, multi-session pool, token refresh
O que e: O modo servidor claude remote-control gerencia um pool de ate 32 child processes concorrentes com tres SpawnModes (single-session, worktree, same-dir), heartbeat JWT, sleep detection e 24h session timeout.
Por que aprender: O standalone bridge e a base de deploys em producao e CI/CD com Claude Code remoto.
Conceitos-chave: BackoffConfig (2s-120s), capacityWake, SessionHandle, BridgeWorkerType (claude_code, claude_code_assistant, cowork).

OAuth Authentication

OAuth 2.0 Authorization Code com PKCE, auto vs manual flow racing, token storage em keychain, refresh com scope expansion e logout seguro.

1
🔐 PKCE Primitives & Sequence
Code verifier, S256 challenge, state CSRF
O que e: PKCE substitui client secret. Code verifier (32 random bytes) vive apenas em memoria. S256 challenge e enviado ao servidor. State parameter protege contra CSRF. Nenhum segredo no binario.
Por que aprender: PKCE e o padrao moderno de seguranca OAuth para CLIs e entender cada valor previne vulnerabilidades.
Conceitos-chave: generateCodeVerifier(), generateCodeChallenge(), generateState(), base64url encoding, S256 method.
2
🏁 Auto vs Manual Flow Racing
Dual flow, localhost callback, paste fallback
O que e: Ambos os flows iniciam simultaneamente. O automatico abre browser com redirect para localhost:PORT. O manual exibe URL para o usuario copiar/colar. Quem entregar o authorization code primeiro vence.
Por que aprender: O racing pattern e elegante e garante que SSH e ambientes headless funcionem sem configuracao extra.
Conceitos-chave: AuthCodeListener (port 0 auto-assign), state validation, pendingResponse pattern, hasPendingResponse().
3
💾 Token Storage & Keychain Architecture
macOS keychain hex encoding, stale-while-error
O que e: Tokens armazenados em keychain nativa (macOS) ou plaintext com 0o600 (Linux). Hex encoding evita shell quoting issues e impede process monitors de ver o token. stdin preferido sobre argv.
Por que aprender: Decisoes de seguranca no armazenamento de credenciais sao criticas e instrutivas para qualquer aplicacao.
Conceitos-chave: security -i stdin vs argv fallback (4032B limit), stale-while-error cache, installOAuthTokens() sequence.
4
🔄 Token Refresh & Profile Skip
5-min buffer, scope expansion, 7M req/day saving
O que e: Refresh proativo com buffer de 5 minutos. Backend permite scope expansion no refresh. Profile fetch e skipped quando todos os campos ja estao cacheados, economizando ~7 milhoes de requests/dia.
Por que aprender: Otimizacoes de token refresh em escala de frota sao um caso de estudo excelente em engenharia de sistemas.
Conceitos-chave: isOAuthTokenExpired(), ALLOWED_SCOPE_EXPANSIONS, profile skip optimization, subscription type mapping.
5
🚪 Logout & Scopes
performLogout() teardown, 6 OAuth scopes
O que e: Logout flush telemetria ANTES de limpar credenciais (previne eventos anonimos). 6 scopes cobrem API key creation, profile, inference, sessions, MCP e file upload. FedStart/FedRAMP endpoints customizaveis.
Por que aprender: O ordering de logout e o modelo de scopes sao padroes de seguranca aplicaveis a qualquer sistema OAuth.
Conceitos-chave: Telemetry-first ordering, clearAuthRelatedCaches(), shouldUseClaudeAIAuth(), ALLOWED_OAUTH_BASE_URLS allowlist.
6
🏢 Enterprise & FedStart Configuration
Custom OAuth URLs, strict allowlist enforcement
O que e: CLAUDE_CODE_CUSTOM_OAUTH_URL redireciona todos os endpoints OAuth para deployments FedRAMP aprovados. Strict allowlist previne exfiltracao de credenciais para servidores arbitrarios.
Por que aprender: Configuracao enterprise demonstra como aplicar controles de seguranca em ambientes regulados.
Conceitos-chave: ALLOWED_OAUTH_BASE_URLS, FedStart beacon/staging endpoints, buildAuthUrl() parameter set, ?code=true upsell flag.

Git Integration

Leitura zero-subprocess do estado Git via filesystem direto, config parser, GitFileWatcher com cache dirty-bit, e operation tracking via regex.

1
📁 Zero-Subprocess Design Philosophy
5 source files, direct .git/ filesystem reads
O que e: Git state (branch, SHA, remote URL) e lido diretamente de .git/ files via Node fs APIs, nunca por subprocesso git. Cinco modulos cobrem filesystem, config parser, gitignore, operation tracking e gh auth.
Por que aprender: Eliminar subprocessos no hot path e uma decisao de performance fundamental com implicacoes de seguranca.
Conceitos-chave: gitFilesystem.ts, gitConfigParser.ts, resolveGitDir (worktree/submodule support), memoized per cwd.
2
📝 Git Config Parser & HEAD Reading
INI format, case rules, ref resolution
O que e: Parser hand-written que replica as regras de git: sections/keys case-insensitive, subsections case-sensitive, escapes dentro de aspas, inline comments. readGitHead parse ref: e SHA detached.
Por que aprender: O parser e verificado contra git source (config.c) e demonstra parsing robusto de formato legado.
Conceitos-chave: Three-level lookup (section, subsection, key), resolveRef (loose + packed-refs + worktree fallback), commonDir.
3
👀 GitFileWatcher: Dirty-Bit Cache
fs.watchFile, lazy recompute, branch switch
O que e: Singleton que mantem branch, HEAD SHA, remote URL e default branch em memoria, recomputando apenas quando arquivos mudam. Dirty flag cleared ANTES do async compute previne race conditions.
Por que aprender: O padrao dirty-before-compute e uma solucao elegante para invalidacao de cache assincrona.
Conceitos-chave: CacheEntry dirty flag, watchCurrentBranchRef(), computeDefaultBranch() 3-step cascade, inotify/kqueue-backed.
4
🛡️ Security: Ref Validation
isSafeRefName, isValidGitSha, injection defense
O que e: Strict allowlist [a-zA-Z0-9/._+@-] para ref names bloqueia path traversal, argument injection e shell metacharacters. Apenas SHAs full-length (40 ou 64 hex) sao aceitos. Falha retorna null silenciosamente.
Por que aprender: Validacao de input de filesystem e uma superficie de ataque real quando .git/ pode ser manipulado.
Conceitos-chave: isSafeRefName regex, isValidGitSha (SHA-1 40 + SHA-256 64), graceful degradation to safe fallback.
5
🔍 Operation Tracking & PR Auto-Linking
Regex parsing, gh/glab/curl detection
O que e: detectGitOperation() parse command text e output via regex shell-agnostic para detectar commits, pushes, merges, rebases e PR creation (gh, glab, curl). PR URLs sao extraidos e linkados a sessao.
Por que aprender: Operation tracking e o bridge entre execucao de comandos e awareness contextual do sistema.
Conceitos-chave: gitCmdRe() tolerant of global flags, PR endpoint regex with sub-resource exclusion, dynamic import for circular deps.
6
🔐 GitHub Auth & Global Gitignore
gh auth token offline check, ~/.config/git/ignore
O que e: gh auth token (nao auth status) verifica autenticacao offline sem network request. stdout: 'ignore' garante que o token nunca entra na memoria Node. Global gitignore em ~/.config/git/ignore mantem .claude/ fora de todos os repos.
Por que aprender: Decisoes de seguranca como stdout: 'ignore' e global gitignore demonstram engenharia defensiva exemplar.
Conceitos-chave: getGhAuthStatus() offline-only, addFileGlobRuleToGitignore() global vs local, isPathGitignored() pre-check.

Upstream Proxy

HTTPS CONNECT proxy MITM-capable via WebSocket tunnel em containers CCR, com protobuf hand-rolled, prctl anti-ptrace e dual authentication.

1
🏗️ Initialization Sequence
6 steps, fails-open design, token lifecycle
O que e: 6 passos sequenciais: guard env vars, read session token, set non-dumpable (prctl), download CA bundle, start relay, unlink token file. Cada step fails-open para nunca quebrar uma sessao funcional.
Por que aprender: O design fails-open com token deletion sequenciada e um padrao de seguranca robusto para sistemas distribuidos.
Conceitos-chave: CLAUDE_CODE_REMOTE + CCR_UPSTREAM_PROXY_ENABLED guards, prctl(PR_SET_DUMPABLE, 0), token deleted AFTER relay confirms.
2
🔀 CONNECT-over-WebSocket Relay
Two-phase state machine, TCP coalescing race
O que e: Relay TCP local recebe CONNECT requests, abre WebSocket para gateway e tunela bytes bidirecionalmente. Two-phase handling: fase 1 acumula CONNECT header, fase 2 pumpa bytes para WS. Race condition de TCP coalescing resolvida via pending buffer.
Por que aprender: O relay demonstra programacao de rede low-level com tratamento correto de races e backpressure.
Conceitos-chave: handleData() two-phase, openTunnel(), st.pending[] for coalesced bytes, st.established TLS corruption guard.
3
📦 Hand-Rolled Protobuf Encoding
Tag 0x0a, varint length, zero-length keepalive
O que e: Encoding manual do UpstreamProxyChunk protobuf: tag byte 0x0a (field 1, wire type 2), varint length, payload. Keepalives sao chunks zero-length. Content-Type: application/proto necessario para o gateway.
Por que aprender: Implementacao protobuf manual demonstra entendimento profundo do wire format e evita dependencia pesada.
Conceitos-chave: encodeChunk()/decodeChunk(), varint encoding, 512KB MAX_CHUNK_BYTES (Envoy cap), 30s ping interval.
4
🛡️ Security Model & Dual Auth
prctl anti-ptrace, WS auth vs CONNECT auth
O que e: prctl(PR_SET_DUMPABLE, 0) bloqueia ptrace same-UID contra prompt injection. Dual auth: Bearer token para WS upgrade + Proxy-Authorization Basic para CONNECT tunnel. TLS corruption guard fecha TCP sem resposta apos established.
Por que aprender: Defesa em camadas contra prompt injection e um modelo de seguranca avancado e instrutivo.
Conceitos-chave: PR_SET_DUMPABLE via Bun FFI, Authorization vs Proxy-Authorization, st.established/st.closed guards.
5
🔧 Bun vs Node Runtime & Backpressure
writeBuf drain queue, WebSocket proxy agent
O que e: Node buffers writes incondicionalmente; Bun faz partial kernel write e silenciosamente descarta o resto. Relay Bun usa writeBuf[] com drain handler. WS proxy agent necessario porque undici nao consulta global dispatcher para upgrades.
Por que aprender: Diferencas de backpressure entre runtimes sao uma classe critica de bugs em sistemas de rede.
Conceitos-chave: Bun sock.write() partial write, writeBuf drain pattern, ws npm package for Node, Bun native proxy URL.
6
🌐 Environment Variable Propagation
8 env vars, NO_PROXY triple format, HTTPS only
O que e: 8 env vars injetadas em todo subprocess: HTTPS_PROXY, SSL_CERT_FILE, NODE_EXTRA_CA_CERTS, REQUESTS_CA_BUNDLE, CURL_CA_BUNDLE etc. NO_PROXY lista Anthropic em 3 formatos para compatibilidade cross-runtime.
Por que aprender: Propagacao correta de proxy config para multiplos runtimes e um problema pratico comum em ambientes enterprise.
Conceitos-chave: HTTPS_PROXY only (no HTTP_PROXY), NO_PROXY triple format, inherited proxy vars for grandchild processes, CA bundle concatenation.

Cron & Scheduling

Scheduler cron built-in com JSON file storage, distributed lock, jitter deterministico por task ID e integracao React via hook.

1
📋 CronTask Data Model
Durable vs session-only, recurring vs one-shot
O que e: CronTask com id (8-hex), cron (5-field local time), prompt, recurring flag e durable flag. Durable tasks persistem em .claude/scheduled_tasks.json. Session-only tasks vivem em memoria e morrem com o processo.
Por que aprender: O modelo de dados dual (durable/session) e o padrao bones/soul demonstram design elegante para diferentes lifecycles.
Conceitos-chave: durable flag never written to disk, permanent flag (exempt from expiry), agentId routing, MAX_JOBS=50 cap.
2
⏱️ Scheduler Lifecycle & Check Loop
1s tick, lazy enable, chokidar watcher
O que e: Scheduler criado uma vez por sessao com design lazy-enable. check() executa a cada 1s, iterando file tasks (se lock owner) e session tasks. Recurring tasks reschedulam de now (nao do fire time) para evitar catch-up rapido.
Por que aprender: O design de catch-up prevention e o lazy-enable pattern sao aplicaveis a qualquer sistema de scheduling.
Conceitos-chave: enablePoll probing, chokidar file watcher, reschedule from now (not fire time), 7-day auto-expiry.
3
🔒 Multi-Session Distributed Lock
PID-based liveness probe, session task exemption
O que e: Lock per-projeto previne double-firing quando multiplas sessoes Claude compartilham diretorio. Non-owner proba a cada 5s para takeover se owner crashar. Session tasks sao lock-exempt (vivem em memoria privada).
Por que aprender: Distributed locking com PID-based liveness e a solucao minima correta para coordenacao local.
Conceitos-chave: tryAcquireSchedulerLock(), LOCK_PROBE_INTERVAL_MS=5000, isOwner gate for file tasks, session tasks unconditional.
4
🎯 Deterministic Jitter System
Forward vs backward jitter, fleet load spreading
O que e: Jitter deterministico baseado no task ID (parseInt hex / 0x100000000). Recurring: forward jitter ate 10% do intervalo (cap 15min). One-shot: backward jitter ate 90s cedo, apenas em :00/:30 (minutos que humanos escolhem).
Por que aprender: O sistema de jitter e um caso de estudo perfeito em engenharia de escala para prevenir thundering herd.
Conceitos-chave: jitterFrac() stable per taskId, recurringFrac=0.1, oneShotMinuteMod=30, GrowthBook live tuning.
5
🛠️ CronCreate/Delete/List Tools
50-job cap, off-minute heuristic, teammate scoping
O que e: CronCreate valida cron e limita a 50 jobs. System prompt instrui modelo a evitar :00/:30 para load shedding. CronList mostra apenas crons do teammate; lead ve tudo. CronDelete valida ownership.
Por que aprender: O off-minute heuristic no system prompt e o maior lever de load shedding, complementando o jitter runtime.
Conceitos-chave: Off-minute heuristic in system prompt, teammate agentId isolation, durable crons blocked for teammates.
6
🔄 REPL Wiring & Missed Tasks
useScheduledTasks hook, startup catch-up, injection defense
O que e: useScheduledTasks hook conecta scheduler ao REPL com priority 'later' (nunca interrompe query ativa). Missed one-shot tasks na startup geram notificacao com adaptive backtick fence (injection defense) e pedem confirmacao.
Por que aprender: A defesa de prompt injection via adaptive backtick fence e a integracao React demonstram engenharia de seguranca full-stack.
Conceitos-chave: WORKLOAD_CRON lower QoS, orphaned teammate cron cleanup, buildMissedTaskNotification() CommonMark fence-length rule.

Voice System

Pipeline push-to-talk com WebSocket STT, multi-backend audio capture (NAPI/arecord/SoX), hold threshold, silent-drop replay e focus mode.

1
🎤 Feature Gating & Audio Backends
Double gate (GrowthBook + OAuth), NAPI/arecord/SoX chain
O que e: Voice e double-gated: GrowthBook VOICE_MODE flag + Anthropic OAuth token. Tres backends em ordem: NAPI (in-process via cpal), arecord (Linux ALSA, 150ms runtime probe), SoX (external process). Probe memoizado.
Por que aprender: O fallback chain com runtime probe (nao apenas PATH check) demonstra deteccao robusta de disponibilidade.
Conceitos-chave: isVoiceGrowthBookEnabled(), hasVoiceAuth(), probeArecord() 150ms race, SoX --buffer 1024 anti-delay.
2
📶 WebSocket STT Protocol
voice_stream endpoint, finalize() 4 triggers
O que e: Conecta a wss://api.anthropic.com/api/ws/speech_to_text/voice_stream com linear16 PCM 16kHz mono. finalize() resolve via 4 caminhos: post_closestream_endpoint (~300ms), no_data_timeout (1.5s), ws_close, safety_timeout (5s).
Por que aprender: O protocolo STT e o finalize() multi-trigger demonstram design resiliente para comunicacao em tempo real.
Conceitos-chave: KeepAlive cada 8s, CloseStream em setTimeout(0), api.anthropic.com (nao claude.ai) por JA3 fingerprint, Buffer.from() anti-race.
3
🤚 Hold-to-Talk Mechanics
5-press threshold, stripTrailing(), release detection
O que e: Terminal nao tem keyup. Hold detectado por timing de auto-repeat (30-80ms gap). 5 presses rapidos ativam voice (previne acidental). Modifier combos ativam no primeiro press. stripTrailing() limpa spaces vazados.
Por que aprender: Reconstruir estado de teclado sem keyup events e um problema classico de terminal UI.
Conceitos-chave: RAPID_KEY_GAP_MS=120, HOLD_THRESHOLD=5, WARMUP_THRESHOLD=2, RELEASE_TIMEOUT_MS=200, full-width space normalization.
4
🔄 Session State Machine & Silent-Drop Replay
idle/recording/processing, 1% bug workaround
O que e: 3 estados: idle, recording, processing. State 'recording' setado SINCRONAMENTE antes de qualquer await. ~1% das sessoes hit silent-drop (CE pod sticky). Client detecta e replaya full audio buffer em novo WS com 250ms backoff.
Por que aprender: O silent-drop replay e um workaround client-side para bug server-side, demonstrando resiliencia na pratica.
Conceitos-chave: Synchronous state transition before await, 6 conditions for replay, 32KB replay slice size, fullAudioRef skipped in focus mode.
5
🎯 Focus Mode & Language/Keyterms
Terminal focus trigger, 5s silence timeout, STT boosting
O que e: Focus mode: recording inicia com focus gain, para com focus loss. Transcripts flushed imediatamente (nao acumulados). 5s silence timeout. Keyterms (ate 50) de termos globais + project + branch boosteiam STT.
Por que aprender: Focus mode demonstra design arquitetural diferente do hold-to-talk com tradeoffs especificos.
Conceitos-chave: FOCUS_SILENCE_TIMEOUT_MS=5000, no audio buffer (no replay), normalizeLanguageForSTT() BCP-47, splitIdentifier() camelCase.
6
📊 Audio Visualization & React Context
RMS computation, sqrt curve, Store-based state
O que e: 16-bar waveform via RMS computation de PCM 16-bit. sqrt(normalized) spreads quieter levels. VoiceState em custom Store (nao React state) via useSyncExternalStore. useVoiceEnabled memoiza auth por authVersion.
Por que aprender: Audio visualization com sqrt curve e o pattern Store + useSyncExternalStore demonstram otimizacao de rendering.
Conceitos-chave: computeLevel() RMS, AUDIO_LEVEL_BARS=16, sqrt curve for speech energy, Store subscription slicing, early-error retry (250ms).
