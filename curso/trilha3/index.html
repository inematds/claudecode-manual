Skills System

Workflows reutilizaveis como slash commands: discovery, parsing, substituicao de argumentos, execucao inline/fork e integracao MCP.

1
⚡ Skill Lifecycle: Six Stages
Discovery, Load, Parse, Substitute, Execute, Inject
O que e: Uma skill e um "named, reusable prompt workflow" armazenado como SKILL.md com YAML frontmatter. Lifecycle em 6 estagios: Discovery (walk directories, resolve symlinks), Load (read + parse YAML), Parse (extract fields, validate hooks), Substitute ($ARGUMENTS, shell backticks), Execute (inline ou forked), Inject (tool result + model overrides).
Por que aprender: Skills sao o mecanismo primario de extensibilidade para workflows customizados. Entender o lifecycle completo permite criar skills poderosas.
Conceitos-chave: SKILL.md format, YAML frontmatter, $ARGUMENTS substitution, inline vs fork context, shell injection (local skills only).
2
📁 Four Sources & Priority Order
managed -> user -> project -> bundled
O que e: Skills carregadas de 4 fontes com prioridade: Managed/Policy (managed/.claude/skills/, enterprise-controlled), User (~/.claude/skills/, pessoal), Project (.claude/skills/, repo-scoped), Bundled (compiladas no CLI). "When two sources define a skill with the same name, the first one loaded wins."
Por que aprender: A prioridade determina qual skill executa quando ha conflito de nomes. Enterprise (managed) sempre vence.
Conceitos-chave: Deduplication by resolved file path (not name), managed = highest priority, bundled = lowest, directory name becomes slash command.
3
📝 SKILL.md Format & Frontmatter
15+ fields: tools, model, effort, paths, hooks
O que e: SKILL.md com YAML frontmatter definindo: name, description, when_to_use, allowed-tools (patterns como Bash(gh:*)), argument-hint, arguments, context (fork), model override, effort level, version, user-invocable, paths (conditional activation), hooks e shell interpreter.
Por que aprender: O frontmatter controla tudo: quais tools a skill pode usar, em qual modelo roda, quando ativa automaticamente. Dominar o formato e essencial.
Conceitos-chave: allowed-tools patterns, context: fork (isolated sub-agent), paths for conditional activation, model override, effort: low/medium/high, disable-model-invocation.
4
🔄 Argument Substitution Pipeline
Named args, indexed args, shell injection, special vars
O que e: Pipeline de substituicao em 6 passos: named args ($foo, $bar) -> indexed args ($ARGUMENTS[0], $0, $1) -> full arg string ($ARGUMENTS) -> fallback append -> shell injection (!`command` ou ```! blocks, local only) -> special variables (${CLAUDE_SKILL_DIR}, ${CLAUDE_SESSION_ID}).
Por que aprender: A substituicao de argumentos e o mecanismo de parametrizacao de skills. Shell injection permite skills dinamicas que leem estado do sistema.
Conceitos-chave: Shell-quote parsing for multi-word args, fallback "ARGUMENTS: ..." append if no placeholder found, MCP skills NEVER execute shell injection (hard security boundary).
5
🔌 MCP Skills & Bundled Skills
Remote skills sem shell, /simplify, /commit, /remember
O que e: MCP skills vem de servers remotos, sem shell injection, sem ${CLAUDE_SKILL_DIR}. Bundled skills compiladas no CLI: /simplify (3 parallel review agents), /commit, /remember, /verify, /debug, /stuck. Reference files extraidos com flags seguros (O_EXCL | O_NOFOLLOW | 0o600).
Por que aprender: MCP skills sao remotas e untrusted - a boundary de seguranca e clara. Bundled skills sao exemplos de patterns avancados como parallel review.
Conceitos-chave: MCP skill registry in AppState.mcp.commands, getAllCommands() merge, registerBundledSkill(), secure nonce directories, /skillify (writes SKILL.md).
6
🔒 Permissions, Auto-Allow & Live Reloading
Safe skills auto-approved, chokidar watch, 300ms debounce
O que e: Waterfall de permissoes: deny rules -> remote canonical auto-allow -> explicit allow -> safe properties check (auto-allow se sem allowed-tools, model override, hooks, paths) -> ask user. Live reloading via chokidar com 300ms debounce. Skill listing capped em 1% do context window.
Por que aprender: Skills simples (informacionais) rodam sem dialogo de permissao. Skills poderosas (com Bash access) requerem aprovacao explicita. Entender a distincao e crucial.
Conceitos-chave: Safe properties check (no tools + no model + no hooks + no paths = auto-allow), Bun stat-polling fallback, ConfigChange hooks on reload, memoization cache clear.

Agent System

Delegacao de trabalho entre instancias Claude: tres tipos de agente, sync vs async, fork path com cache sharing e worktree isolation.

1
🤖 Three Agent Types & Priority
BuiltIn, Custom, Plugin -- policySettings wins
O que e: Tres tipos discriminados por source: BuiltInAgentDefinition (dynamic system prompts, nao overridable por user), CustomAgentDefinition (.claude/agents/*.md ou JSON settings), PluginAgentDefinition (bundled com plugins). Prioridade: built-in -> plugin -> userSettings -> projectSettings -> flagSettings -> policySettings (managed wins).
Por que aprender: Entender tipos e prioridade e essencial para criar custom agents e resolver conflitos de nome.
Conceitos-chave: AgentDefinition discriminated union, getActiveAgentsFromList(), isBuiltInAgent/isCustomAgent/isPluginAgent type guards, legacy name "Task" via aliases.
2
🏃 Built-In Agents Deep Dive
General Purpose, Explore, Plan, Verification, Fork
O que e: 6 built-in agents: General Purpose (all tools), Explore (read-only, haiku/inherit, omitClaudeMd saves ~5-15 Gtok/week), Plan (read-only, inherit), Verification (background:true always async), Fork (experimental, cache-identical tools), StatuslineSetup.
Por que aprender: Cada built-in tem restricoes especificas. Explore usa Haiku para economia. Verification e o unico com background:true hardcoded.
Conceitos-chave: Explore disallowedTools (Agent, Edit, Write), omitClaudeMd optimization, verification ephemeral /tmp scripts, custom agent markdown format with frontmatter.
3
🔄 Sync vs Async Lifecycle
shouldRunAsync boolean, 6 conditions, detached lifecycle
O que e: Um unico boolean shouldRunAsync decide tudo. 6 condicoes: run_in_background=true, agent.background=true, isCoordinator, forceAsync, assistantForceAsync, proactive module active. Sync: bloqueia parent turn com await runAgent(). Async: registerAsyncAgent() retorna imediatamente, notifica via enqueueAgentNotification().
Por que aprender: Async agents tem AbortController proprio nao linkado ao parent -- sobrevivem ESC. In-process teammates NAO podem lancar background agents.
Conceitos-chave: shouldRunAsync 6 conditions, isBackgroundTasksDisabled guard, registerAsyncAgent() in AppState, independent AbortController for async, in-process limitation.
4
🔀 The Fork Path & Cache Sharing
buildForkedMessages(), byte-identical prefix, recursive guard
O que e: Fork path (gate: FORK_SUBAGENT) herda full conversation context. buildForkedMessages() cria request byte-identical ate a directiva per-child para maximo cache sharing. Tool_result placeholders identicos: "Fork started -- processing in background". buildChildMessage() com regras non-negotiable.
Por que aprender: Fork e o mecanismo de maximo aproveitamento de prompt cache. Entender por que placeholders devem ser identicos e essencial.
Conceitos-chave: Byte-identical request prefix, placeholder "Fork started", buildChildMessage() STOP-READ-THIS format, recursive fork guard, fork + worktree notice.
5
🌳 Worktree Isolation
Git worktree temporario, smart cleanup
O que e: isolation: 'worktree' cria git worktree temporario antes de spawnar o agent. Smart cleanup: sem git-tracked changes = branch deletado; com changes = branch mantido para parent inspecionar/mergear.
Por que aprender: Worktree isolation permite trabalho paralelo sem conflitos de arquivo. O smart cleanup previne branches orfas.
Conceitos-chave: git worktree add, smart cleanup (no changes = delete, changes = keep), fork + worktree combination, buildWorktreeNotice().
6
📨 SendMessageTool & Agent Frontmatter
Inter-agent messaging, 15+ frontmatter fields
O que e: SendMessageTool (wire: SendMessage) e o primitivo de messaging inter-agente. Routing: "teammate-name" (direct), "*" (broadcast), "uds:path" (cross-session), "bridge:session-id" (remote). 15+ frontmatter fields para custom agents incluindo model, tools, permissionMode, maxTurns, memory, isolation, mcpServers.
Por que aprender: SendMessage conecta agentes em swarms. Os frontmatter fields permitem agents altamente customizados para tarefas especificas.
Conceitos-chave: 4 message types (plain, shutdown_request, shutdown_response, plan_approval_response), 3 addressing schemes, permissionMode: bubble (escalate to parent), requiredMcpServers field.

Coordinator Mode

Modo de operacao onde Claude atua exclusivamente como dispatcher: 4 tools, workflow de 4 fases e a arte de synthesis.

1
📋 Core Concept & Activation
CLAUDE_CODE_COORDINATOR_MODE=1, matchSessionMode()
O que e: Coordinator mode: Claude nunca executa tools diretamente, apenas despacha para worker agents. Ativado via CLAUDE_CODE_COORDINATOR_MODE=1. isCoordinatorMode() le env var sem cache (permite mudanca em sessoes resumidas). matchSessionMode() restaura modo correto em sessoes resumidas.
Por que aprender: Coordinator mode e o padrao para tarefas complexas que beneficiam de paralelismo e separacao de concerns.
Conceitos-chave: No caching (runtime read), matchSessionMode() mutates process.env, exactly 4 tools (Agent, SendMessage, TaskStop, SyntheticOutput).
2
🎯 Four-Phase Task Workflow
Research -> Synthesis -> Implementation -> Verification
O que e: Workflow em 4 fases: Research (workers paralelos investigam codebase), Synthesis (coordinator cria specs com file paths e line numbers - NUNCA "based on your findings"), Implementation (workers fazem changes, rodam tests, commit), Verification (workers provam que codigo funciona independentemente).
Por que aprender: Synthesis e o trabalho mais importante do coordinator. Delegar entendimento resulta em workers sem contexto suficiente.
Conceitos-chave: "Never write 'based on your findings'", coordinator synthesizes specific file paths + line numbers, research parallel / implementation serial per file set.
3
🔄 Continue vs Spawn Fresh Decision
6 situacoes, context overlap como criterio
O que e: 6 regras de decisao: Continue se research explorou exatos files para edit (worker tem context), spawn fresh se research broad mas implementation narrow (evita noise), continue para correcao de falha (worker tem error context), spawn fresh para verificacao (fresh eyes), spawn fresh se wrong approach (context pollutes), spawn fresh para task unrelated.
Por que aprender: A decisao continue vs spawn fresh impacta diretamente qualidade e eficiencia. Context overlap e o criterio chave.
Conceitos-chave: Context overlap analysis, wrong-approach pollution, verifier needs fresh eyes, error context retention value, unrelated = no useful context.
4
💬 Worker Prompt Guidelines
Self-contained prompts com file paths e line numbers
O que e: Bad prompts delegam entendimento: "Fix the bug we discussed". Good prompts sao self-contained: "Fix null pointer in src/auth/validate.ts:42. The user field is undefined when Session.expired is true. Add null check before user.id access -- return 401. Run tests, commit, report hash."
Por que aprender: A qualidade do prompt do worker determina o sucesso da task. Workers nao tem contexto do coordinator -- tudo deve estar no prompt.
Conceitos-chave: Self-contained prompts, specific file paths + line numbers, explicit expected behavior, test command, commit instruction, report format.
5
🛠️ Tool Restrictions & Worker Context
4 coordinator tools, ASYNC_AGENT_ALLOWED_TOOLS
O que e: Coordinator: exatamente 4 tools (Agent, SendMessage, TaskStop, SyntheticOutput). Workers: full ASYNC_AGENT_ALLOWED_TOOLS (Bash, Read, Edit, Glob, Grep, WebSearch, Skill, Notebook, Worktree) + MCP tools. Simple mode restringe workers a Bash, Read, Edit.
Por que aprender: A assimetria de tools e intencional: coordinator planeja, workers executam. Entender isso previne tentativas de filesystem access pelo coordinator.
Conceitos-chave: COORDINATOR_MODE_ALLOWED_TOOLS, ASYNC_AGENT_ALLOWED_TOOLS, getCoordinatorUserContext() (tool enum + MCP + scratchpad), CLAUDE_CODE_SIMPLE worker restriction.
6
📝 The Scratchpad
Cross-worker knowledge, permission-free shared directory
O que e: Scratchpad e um diretorio compartilhado onde workers podem ler e escrever sem permission prompts. Permite durable findings que sobrevivem entre workers para cross-worker knowledge sharing.
Por que aprender: O scratchpad e o mecanismo de comunicacao duravel entre workers. Sem ele, findings de research se perdem quando workers terminam.
Conceitos-chave: Permission-free read/write, cross-worker durability, injected via getCoordinatorUserContext(), shared workspace pattern.

Teams & Swarm

Equipes nomeadas de Claude agents com config compartilhada, messaging file-based, tres spawn backends e lifecycle completo.

1
👥 TeamCreateTool & File Structure
5 sync steps, ~/.claude/teams/ layout, agentName@teamName
O que e: TeamCreateTool executa 5 passos sincronos: uniqueness check (generateWordSlug se nome existe), write TeamFile JSON, create task list, update AppState.teamContext, register session cleanup. Estrutura: ~/.claude/teams/<name>/config.json + permissions/ + tasks/. Agent IDs: agentName@teamName.
Por que aprender: Teams sao a unidade de organizacao para trabalho paralelo com multiplos agents. Entender a estrutura on-disk ajuda debugging.
Conceitos-chave: generateWordSlug() for unique names, TeamFile JSON, task numbering starts at 1, session cleanup hook for orphaned directories.
2
💻 Three Spawn Backends
tmux, iTerm2, in-process -- detection at module load
O que e: Tres backends com prioridade: tmux (highest, kill-pane), iTerm2 (it2 CLI, session close), in-process (AbortController). Detection via env vars capturadas em module load time: TMUX, TMUX_PANE, TERM_PROGRAM, ITERM_SESSION_ID. iTerm2 check usa 'it2 session list' (nao --version).
Por que aprender: O backend determina como teammates sao spawned e killed. In-process teammates compartilham API client e MCP connections mas tem AbortController independente.
Conceitos-chave: tmux kill-pane, it2 session close, in-process AsyncLocalStorage isolation, shared API client, independent AbortController (not linked to leader).
3
📬 Mailbox Messaging System
File-based JSON, 6 message types, inbox per agent
O que e: Comunicacao via mailbox file-based: ~/.claude/teams/<team>/inbox/<agent-name>/. 6 tipos de mensagem: plain text, shutdown_request (lead -> teammate), idle notification (system-generated), permission_request (worker -> lead), permission_response (lead -> worker), mode_set_request.
Por que aprender: Todo messaging entre agents e file-based (excecao: leader permission bridge usa module-level setter). Entender os tipos previne deadlocks de comunicacao.
Conceitos-chave: File-based messaging, idle != dead (isActive:false means turn ended, wakes on new message), shutdown_request graceful flow, permission escalation to lead.
4
🔒 Permission Sync
Worker escalation, leader approval/denial
O que e: Workers podem escalar permission requests para o team lead via mailbox. O leader ve o request na UI e aprova ou nega. Isso permite controle centralizado de permissoes em swarms.
Por que aprender: Permission sync e o mecanismo de seguranca para swarms. Sem ele, cada worker precisaria de interacao direta com usuario.
Conceitos-chave: permission_request message type, leader UI approval, permission_response back to worker, centralized permission control.
5
🔄 Full Lifecycle: 4 Phases
Setup -> Parallel Work -> Shutdown -> Cleanup Guard
O que e: Phase 1 Setup: lead cria team, tasks, spawna teammates. Phase 2 Parallel Work: teammates claim tasks, enviam progresso. Phase 3 Shutdown: lead envia shutdown_request, teammates aprovam e saem. Phase 4 Cleanup Guard: se sessao termina sem TeamDelete, shutdown hook limpa panes e diretorios orfaos.
Por que aprender: O lifecycle completo previne orphaned processes e directories. TeamDelete blocks on active members -- sempre shutdown primeiro.
Conceitos-chave: TeamDelete blocks on active, shutdown_request -> approve -> gracefulShutdown(0), session cleanup safety net, CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1 gate.
6
🔑 Key Design Invariants
Team = TaskList, idle != dead, all messaging file-based
O que e: Invariantes criticos: Team = TaskList (TeamCreate cria task directory). Tres backends, uma interface. Detection capturada em module load (env vars read at import). All messaging file-based. Idle nao e dead (isActive:false = turn ended, wakes on message).
Por que aprender: Estes invariantes sao as regras fundamentais do sistema. Violar qualquer um causa bugs sutis e dificeis de debugar.
Conceitos-chave: Team = TaskList coupling, module-load detection (immutable after import), file-based exception (leader permission bridge), TeamDelete blocks on active members.

Memory System

Tres camadas de memoria persistente: Auto Memory, Session Memory e Team Memory com sync server-side e secret scanning.

1
🧠 Three Memory Layers
Auto Memory, Session Memory, Team Memory
O que e: Layer 1 Auto Memory: ~/.claude/projects/<slug>/memory/, fatos persistentes sobre usuario, sobrevive todas sessoes. Layer 2 Session Memory: ~/.claude/session-memory/<uuid>.md, notas in-session para context compaction. Layer 3 Team Memory: .../memory/team/, sync server-side per GitHub repo.
Por que aprender: Memoria e o que faz Claude Code melhorar com uso. Entender as tres camadas permite otimizar o que e lembrado e compartilhado.
Conceitos-chave: Auto = cross-session persistence, Session = single-session compaction, Team = server-synced per repo, MEMORY.md index always in context.
2
📂 MEMORY.md & Topic Files
200 lines/25KB index, YAML frontmatter, 4 types
O que e: MEMORY.md: index de ate 200 linhas / 25KB, funciona como pointer list (nao content storage), sempre injetado no system prompt. Topic files com YAML frontmatter: name, description, type. 4 tipos: user (role, goals), feedback (corrections + confirmations + Why), project (ongoing work, absolute dates), reference (external pointers).
Por que aprender: A taxonomia de 4 tipos com validacao em parse time garante qualidade de memoria. Saber o que NAO salvar e tao importante quanto saber o que salvar.
Conceitos-chave: Pointer list only (not content storage), 4-type taxonomy with parse-time validation, exclusions (code patterns, git history, fix recipes, CLAUDE.md content).
3
🔍 Extraction Pipeline & Relevance Recall
Post-query extraction, Sonnet selector, staleness signals
O que e: Extraction roda APOS query loops completos, nunca durante conversas. Mutual exclusion com main agent via hasMemoryWritesSince(). Relevance Recall: Sonnet side-query seleciona ate 5 topic files relevantes, le apenas primeiras 30 linhas (frontmatter range). Staleness: "This memory is N days old."
Por que aprender: Extraction post-query e mutual exclusion previnem conflitos. O selector de 5 files mantém context window gerenciavel.
Conceitos-chave: hasMemoryWritesSince() mutex, Sonnet selector max 5 files, 30-line frontmatter read, staleness injection, point-in-time observation warning.
4
📝 Session Memory Structure
10 fixed sections, extraction triggers, 12K token budget
O que e: 10 secoes fixas: Session Title, Current State, Task specification, Files and Functions, Workflow, Errors & Corrections, Codebase Documentation, Learnings, Key results, Worklog. Triggers: 10K tokens para init, 5K entre updates, 3 tool calls entre updates. Budget: 12K total, 2K per section.
Por que aprender: Session memory powers context compaction. Os triggers balanceiam freshness vs overhead de extraction.
Conceitos-chave: Fixed 10-section structure, minimumMessageTokensToInit: 10K, minimumTokensBetweenUpdate: 5K, toolCallsBetweenUpdates: 3, 12K total / 2K per section budget.
5
🌐 Team Memory Sync
Delta push with SHA-256, secret scanning, scope routing
O que e: Server-synced subdirectory gated por TEAMMEM flag + OAuth. Sync contract: pull (server wins per-key), push (delta upload via SHA-256 hash comparison, batched 200KB max). File deletions nao propagam. 35+ secret patterns do gitleaks ruleset bloqueiam push -- "secrets never reach the server."
Por que aprender: Team memory permite compartilhar conhecimento entre membros do time. Secret scanning e a guardrail critica contra vazamento de credenciais.
Conceitos-chave: Delta push (SHA-256), server wins per-key on pull, 200KB batch limit, 35+ gitleaks patterns, user type always private, project type bias toward team.
6
⚙️ Path Resolution & Disable Mechanisms
3-tier resolution, security boundary, 4 disable flags
O que e: Path resolution: CLAUDE_COWORK_MEMORY_PATH_OVERRIDE (env) -> autoMemoryDirectory (settings, trusted sources only) -> default (~/.claude/projects/<slug>/memory/). Project settings excluded from override (previne repos maliciosos redirecionando para ~/.ssh). 4 disable flags: CLAUDE_CODE_DISABLE_AUTO_MEMORY, CLAUDE_CODE_SIMPLE, autoMemoryEnabled:false, tengu_passport_quail.
Por que aprender: A exclusao de project settings do override e uma decisao de seguranca critica. Saber desabilitar memoria e essencial para debugging.
Conceitos-chave: Trusted sources only for path override, project settings excluded (security), sanitized-git-root slug, 4 independent disable mechanisms.

Auto-Memory & Dreams

Dois processos background: extraction per-turn e consolidation cross-session (Dream). Recall seletivo via Sonnet e o lock file elegante.

1
📁 Memory Directory & Four Types
Resolution order, MEMORY.md index, topic file YAML
O que e: Directory resolution: env override -> settings.json -> ~/.claude/projects/<slug>/memory/. Contem MEMORY.md (index, always in system prompt), topic files (user_role.md, feedback_testing.md, etc), .consolidate-lock e logs/. 4 tipos: user (profile), feedback (corrections + confirmations), project (ongoing work), reference (external pointers).
Por que aprender: A estrutura de diretorio e o foundation de toda a memoria persistente. Os 4 tipos com YAML frontmatter garantem organizacao semantica.
Conceitos-chave: MEMORY.md always injected, topic files loaded on-demand by Sonnet, .consolidate-lock mtime IS lastConsolidatedAt, logs/ for KAIROS/assistant mode.
2
⚡ Layer 1: Extract Memories
Per-turn extraction, forked agent, mutual exclusion
O que e: handleStopHooks fires executeExtractMemories() apos cada query. Forked agent pattern compartilha prompt cache do parent. Closure-scoped state: lastMemoryMessageUuid (cursor), inProgress (overlap guard), turnsSinceLastExtraction (throttle). Mutual exclusion via hasMemoryWritesSince().
Por que aprender: Extraction e o mecanismo que popula Auto Memory automaticamente. O forked agent pattern com cache sharing e reusavel em outros contextos.
Conceitos-chave: Forked agent shares parent prompt cache, two-turn strategy (Turn 1: parallel Reads, Turn 2: parallel Writes), tool permissions: read-only + auto-mem writes only.
3
💭 Layer 2: Auto Dream (Consolidation)
3 gates, 4 phases, lock file mtime trick
O que e: Auto Dream fires via executeAutoDream() quando 3 gates passam: Time Gate (24h desde ultima consolidacao), Session Gate (5+ transcripts), Lock Gate (nenhum outro processo consolidando). Lock file .consolidate-lock: body = PID, mtime IS lastConsolidatedAt (1 stat() read). 4 phases: Orient, Gather recent signal, Consolidate (merge/prune/convert dates), Prune and index.
Por que aprender: Dream e a "manutencao" da memoria: merge duplicatas, converte datas relativas em absolutas, mantem MEMORY.md sob 200 linhas. O lock file mtime trick e elegante.
Conceitos-chave: 3 gates (time 24h, sessions 5+, lock), lock file mtime = lastConsolidatedAt, 4 consolidation phases, relative date conversion, 200 lines/25KB MEMORY.md limit.
4
🔍 Recall: findRelevantMemories
Sonnet side-query, max 5 files, tool suppression
O que e: Em query time, Sonnet side-query seleciona ate 5 topic files relevantes. Excecao: se modelo esta usando tool ativamente (e.g. mcp__X__spawn em recentTools), reference documentation desse tool e suprimida. Staleness: "This memory is 47 days old. Memories are point-in-time observations, not live state."
Por que aprender: Recall seletivo mantem context window gerenciavel. Tool suppression previne informacao redundante quando tool docs ja estao no contexto.
Conceitos-chave: Sonnet selector max 5 files, recentTools suppression, staleness injection, "Before recommending from memory" section header (outperformed abstract variants).
5
🌐 Team Memory Scope Routing
Private vs team, prompt-guided enforcement
O que e: Routing per type: user = always private, feedback = private by default (team only for project-wide conventions), project = strongly bias toward team, reference = usually team. Enforcement via prompt guidance, nao filesystem permissions.
Por que aprender: Scope routing determina o que e compartilhado com o time. User memories nunca devem ser team (informacao pessoal).
Conceitos-chave: Prompt-guided enforcement (not filesystem), user always private, feedback default private, project bias team, TEAMMEM flag gate.
6
🌌 KAIROS Mode & Cache Efficiency
Append-only daily logs, /dream skill, prompt cache sharing
O que e: KAIROS/assistant mode: sessoes longas usam append-only daily logs em logs/YYYY/MM/YYYY-MM-DD.md. /dream skill distila logs em topic files. Cache efficiency: forked agent compartilha prompt cache do parent. Tool list deve match main agent para cache sharing (tools sao parte da cache key).
Por que aprender: KAIROS e o modo para assistentes persistentes. Cache efficiency e critica para manter custos baixos em extraction/dream.
Conceitos-chave: Append-only daily logs, /dream nightly distillation, forked agent prompt cache sharing, tool list = cache key, extraction agent runs after query loop.
