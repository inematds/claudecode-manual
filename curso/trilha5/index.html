Permissions

O pipeline de decisao de permissoes: 5 modos, rule matching, AI classifier, shadowed rule detection e permission explainer.

1
🛡️ Decision Pipeline (Steps 1a-3)
Deny first, safety checks, mode transformations
O que e: Pipeline em hasPermissionsToUseToolInner(): 1a deny rule, 1b ask rule, 1c tool checkPermissions(), 1e requiresUserInteraction, 1f content-specific ask, 1g safety check (.git/, .claude/, .vscode/). 2a bypass mode. 2b tool-wide allow. 3 mode transformations.
Por que aprender: O pipeline de permissoes e a camada de seguranca mais critica do Claude Code. Entender a ordem e essencial.
Conceitos-chave: Deny rules checked first, bypassPermissions cannot bypass deny/safety/requiresUserInteraction, 3 outcomes (allow/deny/ask).
2
🔒 The 5 Permission Modes
default, plan, acceptEdits, bypass, dontAsk, auto
O que e: default (prompt tudo), plan (read-only), acceptEdits (auto-approve file edits), bypassPermissions (skip prompts exceto deny/safety), dontAsk (ask->deny), auto (AI classifier decide).
Por que aprender: Cada modo define o trade-off entre autonomia do agente e controle do usuario.
Conceitos-chave: 5 modes + auto (ANT-only), bypassPermissions exceptions, dontAsk converts ask->deny, auto routes through classifier.
3
📋 Rule Matching System
exact, prefix (legacy), wildcard, MCP rules
O que e: Regras como Bash, Bash(npm install) (exact), Bash(npm:*) (legacy prefix), Bash(git add *) (wildcard). Trailing * com espaco faz match opcional. Legacy tool name aliases auto-migrados (Task->Agent, KillShell->TaskStop).
Por que aprender: Entender a sintaxe de regras e crucial para configurar permissoes corretamente sem regras shadowed.
Conceitos-chave: 3 content rule types (exact/prefix/wildcard), trailing * optional match, escaped asterisk, legacy tool name migration.
4
🤖 Auto Mode & AI Classifier
Fast paths, safe allowlist, denial limits
O que e: 3 fast paths: non-classifiable safety checks, acceptEdits fast path, safe tool allowlist. Denial tracking: 3 consecutivas ou 20 totais -> fallback para prompting. Dangerous patterns (python, node, sudo) stripped ao entrar em auto mode.
Por que aprender: O classifier demonstra como AI pode ser usada para decisoes de seguranca com guardrails robustos.
Conceitos-chave: Layered fast paths, SAFE_YOLO_ALLOWLISTED_TOOLS, 3/20 denial limits, DANGEROUS_BASH_PATTERNS stripping, fail closed by default.
5
🔍 Shadowed Rule Detection
Deny-shadowed, ask-shadowed, unreachable rules
O que e: Uma regra pode ser unreachable: Bash em deny list torna Bash(ls:*) em allow list nunca avaliada (deny-shadowed). Bash em ask list torna allow rules unreachable (ask-shadowed). Detector reporta conflitos.
Por que aprender: Regras shadowed sao uma classe comum de bugs de configuracao que silenciosamente invalidam permissoes.
Conceitos-chave: Deny-shadowed (tool-wide deny blocks specific allow), ask-shadowed (tool-wide ask blocks specific allow), sandbox auto-allow exception.
6
💡 Permission Explainer & Rule Sources
Side API call, 8 rule sources, priority
O que e: Explainer faz side API call para gerar explicacao human-readable com risk level (LOW/MEDIUM/HIGH). 8 fontes de regras: policySettings, projectSettings, userSettings, localSettings, flagSettings, cliArg, session, command.
Por que aprender: A hierarquia de fontes determina quais regras vencem e como enterprise pode lockdown configuracoes.
Conceitos-chave: sideQuery() (separate API call), 8 rule sources, allowManagedPermissionRulesOnly (enterprise lockdown), structured output schema.

Settings & Config

Cascade de 5 camadas, merge semantics, 3-tier cache, change detection e remote managed settings.

1
📚 5-Layer Settings Cascade
user -> project -> local -> flag -> policy
O que e: 5 camadas em prioridade crescente: userSettings, projectSettings, localSettings, flagSettings, policySettings. Arrays sao concatenados e deduplicados. Scalars: camada superior vence. policySettings usa first-source-wins internamente.
Por que aprender: A cascade determina qual configuracao o usuario efetivamente ve e e o fundamento de todo o sistema de settings.
Conceitos-chave: 5 layers, array dedup-concatenation, scalar override, policySettings first-source-wins (remote > MDM > file > HKCU).
2
🔄 Merge Semantics & 3-Tier Cache
settingsMergeCustomizer, clone-on-read, fanOut
O que e: lodash mergeWith() com customizer: arrays concatenados+deduplicated, objetos deep-merged, scalars override. 3-tier cache: sessionSettingsCache, perSourceCache, parseFileCache. Todos invalidados atomicamente por resetSettingsCache(). Clone-on-read previne mutacao.
Por que aprender: O cache de 3 niveis e a razao pela qual startup e rapido. Entender invalidacao atomica previne bugs de stale config.
Conceitos-chave: settingsMergeCustomizer, uniq([...a, ...b]), 3-tier cache with atomic invalidation, clone-on-read, fanOut() single-producer pattern.
3
🔍 Change Detection & Hot-Reload
chokidar, internal write suppression, MDM poll
O que e: chokidar com 500ms awaitWriteFinish. markInternalWrite()/consumeInternalWrite() com janela de 5s para suprimir self-triggered loops. Delete-and-recreate grace period de 1700ms. MDM poll a cada 30 minutos.
Por que aprender: Change detection e a maquinaria que permite hot-reload de configuracoes sem restart.
Conceitos-chave: FILE_STABILITY_THRESHOLD_MS=1000, INTERNAL_WRITE_WINDOW_MS=5000, DELETION_GRACE_MS=1700, MDM_POLL_INTERVAL_MS=30min.
4
🌐 Remote Managed Settings
ETag caching, fail-open, security check
O que e: Enterprise/Team: fetch da API Anthropic no startup com ETag (SHA-256 checksum). Fail-open: fetch falha com cache -> usa stale; sem cache -> prossegue sem. Background polling a cada 60 min. Security check para changes perigosos (hooks, permission rules).
Por que aprender: Remote settings e o mecanismo de controle enterprise mais poderoso, com fail-open por design.
Conceitos-chave: SHA-256 ETag, fail-open design, 5 retries with exponential backoff, checkManagedSettingsSecurity(), 60min polling.
5
🔒 Security Constraints
projectSettings exclusion, managed-only locks
O que e: skipDangerousModePermissionPrompt, skipAutoPermissionPrompt, useAutoModeDuringPlan excluem projectSettings do trust domain (previne RCE via repo malicioso). allowManagedPermissionRulesOnly ignora todas as regras nao-managed.
Por que aprender: Entender quais settings sao excluidas de projectSettings previne vetores de ataque via repositorios.
Conceitos-chave: projectSettings trust exclusion, allowManagedPermissionRulesOnly, allowManagedHooksOnly, RCE prevention.
6
💾 Settings Sync & Backward Compatibility
CCR sync, schema contract, drop-in directory
O que e: Settings sync entre CLI e CCR via cloud key-value store (scoped por git remote hash). Contrato: proibido remover campos, enum values, tornar optional required. managed-settings.d/ com merge alfabetico (convencao systemd).
Por que aprender: Backward compatibility e o que garante que updates nunca quebram configuracoes existentes.
Conceitos-chave: Upload incremental (diff-only PUT), 500KB max file size, alphabetical drop-in merge, .passthrough() on permissions, filterInvalidPermissionRules().

Session Management

Persistencia JSONL, linked list via parentUuid, state machine tri-state, interrupt detection e cloud sync.

1
💾 JSONL Storage & Write Queue
Append-only, batched writes, lazy materialization
O que e: Sessions como JSONL em ~/.claude/projects/. Project singleton com write queue batched (100ms local, 10ms cloud). Session file criado lazy no primeiro user/assistant message. Dedup via messageSet Set<UUID>.
Por que aprender: O design append-only com batched writes e o padrao para persistencia de alta throughput.
Conceitos-chave: Append-only JSONL, Project singleton, lazy materialization, UUID deduplication, 100ms/10ms drain intervals.
2
🔗 parentUuid Linked List
Chain walk, branching, compact boundary
O que e: Mensagens formam linked list via parentUuid. Load: walk da folha mais nova ate root, depois reverse. Sidechains para subagents em arquivos separados. Compact boundary: parentUuid=null trunca a chain logica.
Por que aprender: O parentUuid design permite branching e forking enquanto mantendo JSONL append-only.
Conceitos-chave: Singly-linked list (stored reversed), buildConversationChain(), cycle detection, sidechain exception, compact boundary null parent.
3
🚦 Session State Machine
idle/running/requires_action, CCR bridge
O que e: Tri-state: idle, running, requires_action. Transicoes disparam callbacks para CCR bridge, SDK event stream e metadata store. pending_action mostra o que o modelo esta bloqueado sem mudanca de schema protobuf.
Por que aprender: A state machine e o choke point que conecta o estado da sessao a todas as interfaces externas.
Conceitos-chave: notifySessionStateChanged() choke point, RequiresActionDetails, JSON Merge Patch for external metadata, task_summary for long turns.
4
🔄 Interrupt Detection & Recovery
interrupted_prompt, interrupted_turn, synthetic messages
O que e: Classifica transcript em: none (completo), interrupted_prompt (user enviou, AI nao respondeu), interrupted_turn (AI mid-tool-use). interrupted_turn -> injeta "Continue from where you left off." (isMeta:true).
Por que aprender: Recovery automatico e o que torna resume confiavel sem intervencao manual.
Conceitos-chave: detectTurnInterruption(), synthetic continuation message, isTerminalToolResult (brief mode), NO_RESPONSE_REQUESTED sentinel.
5
☁️ Cloud Sync & Resume Orchestrator
CCR v2 hydration, worktree resume, skill restore
O que e: Dual remote: CCR v2 (internalEventWriter) e v1 Session Ingress (REST). Hydration reescreve JSONL local com dados do servidor. Resume: switchSession, worktree chdir (TOCTOU-safe), skill state restoration, agent setting restore.
Por que aprender: Cloud sync e a base para sessoes cross-device e o CCR web/mobile UI.
Conceitos-chave: Dual persistence paths, hydrateFromCCRv2(), processResumedConversation() 10-step orchestrator, TOCTOU-safe chdir.
6
📋 Session Listing & Tail Window
Lite reads, 64KB tail, reAppendSessionMetadata
O que e: Resume picker usa lite reads: head (4KB) + tail (64KB) por arquivo. Metadata (title, tag, lastPrompt) re-appendada ao EOF em exit/compaction para ficar na tail window. Sem isso, sessions longas mostram "No prompt".
Por que aprender: O tail-window strategy e a razao pela qual session listing e rapido mesmo com arquivos de GB.
Conceitos-chave: readHeadAndTail(), LITE_READ_BUF_SIZE=65536, reAppendSessionMetadata(), metadata in tail window guarantee.

Context Compaction

4 estrategias de compactacao: microcompact, session memory, full LLM summary (9 secoes) e reactive compact.

1
📊 Thresholds & Token Warning States
Effective window, 20k reserve, circuit breaker
O que e: Effective window = raw - 20k (p99.99 de output do compact). 4 estados: Normal (>20k), Warning/Error (<=20k), Auto-Compact (<=13k), Blocking (<=3k). Circuit breaker: 3 failures consecutivas param auto-compact (fix salvou 250K API calls/dia).
Por que aprender: Entender thresholds e o que permite prever quando compaction dispara e por que.
Conceitos-chave: 20k reserve (p99.99), AUTOCOMPACT_BUFFER=13k, MANUAL_COMPACT_BUFFER=3k, MAX_CONSECUTIVE_FAILURES=3.
2
⚡ Microcompact - Zero API Cost
Time-based, cached MC, token estimation
O que e: Content-clear de tool results antigos em memoria (zero API call). Time-based: gap >60min -> clear. Cached MC: queue cache_edits no API layer sem mutar messages (preserva prompt cache). COMPACTABLE_TOOLS: FileRead, Bash, Grep, Glob, etc.
Por que aprender: Microcompact e gratis e sempre roda primeiro. Entender os dois paths (mutate vs cache_edits) e key.
Conceitos-chave: Time-based MC (60min gap), cached MC (cache_edits), keepRecent=5, 4/3 token estimation padding, IMAGE_MAX_TOKEN_SIZE=2000.
3
📝 Session Memory & Full LLM Compact
Zero-call summary, 9-section prompt, analysis scratchpad
O que e: Session memory: usa arquivo de memoria atualizado continuamente (zero API call). Full LLM: fork de sub-agent com prompt de 9 secoes (incluindo "All User Messages" verbatim). analysis block stripped antes de entrar no contexto.
Por que aprender: A escada de custo (microcompact -> session memory -> full LLM) mostra o tradeoff fidelidade vs custo.
Conceitos-chave: Session memory (min 10k/max 40k tokens), 9-section summary, section 6 (user messages verbatim), NO_TOOLS_PREAMBLE, analysis scratchpad.
4
🚨 Reactive Compact & Context Collapse
413 error handler, API-round grouping
O que e: Reactive: triggered por 413 prompt-too-long. Peel API-round groups da ponta mais velha. Context collapse: sistema proprio com thresholds 90%/95%, suprime auto-compact. Grouping por assistant message.id boundaries.
Por que aprender: Reactive compact e o escape hatch quando proactive falha. Context collapse e o futuro do gerenciamento de contexto.
Conceitos-chave: 413 trigger, API-round grouping, CONTEXT_COLLAPSE feature, 90%/95% thresholds, recursion guards (session_memory/compact).
5
🧹 Post-Compact Cleanup
Cache invalidation, subagent safety, file restoration
O que e: runPostCompactCleanup() invalida caches (getUserContext, memoryFiles, classifierApprovals). isMainThread guard previne subagents de corromper caches do main thread. Post-compact restaura ate 5 files (50k token budget, 5k per file).
Por que aprender: Cleanup correto e o que faz compaction funcionar sem efeitos colaterais inesperados.
Conceitos-chave: Centralized cleanup, isMainThread guard, POST_COMPACT_MAX_FILES=5, sentSkillNames intentionally not cleared.
6
📊 /context Command & Accurate Token Display
Pre-API transforms, microcompact applied
O que e: /context aplica as mesmas transformacoes pre-API que query.ts: slice apos compact boundary, context-collapse projection, microcompact. Sem isso, o display mostraria "180k, 3 spans collapsed" quando a API ve 120k.
Por que aprender: Mostra a importancia de replicar o pipeline completo para displays precisos de token count.
Conceitos-chave: toApiView() mirrors query.ts pipeline, getMessagesAfterCompactBoundary(), projectView() for collapse, accurate display.

1
📨 Pre-Sink Queue & Sink Routing
Zero-dep API, sampling, PII separation

Por que aprender: Zero-dep analytics API permite qualquer modulo logar sem criar import cycles.

2
📊 Datadog Transport & Cardinality
Batch flush, model name normalization, user buckets
O que e: Batch de 100 entries ou flush a cada 15s. Allowed events only. Cardinality reduction: model names canonicalizados, MCP tool names -> "mcp", version strings stripped, user IDs hashed para 30 buckets (SHA-256 mod 30).
Por que aprender: Cardinality reduction e essencial para manter Datadog usavel e economico em escala.
Conceitos-chave: MAX_BATCH_SIZE=100, 15s flush, DATADOG_ALLOWED_EVENTS, 30-bucket user hashing, .unref() timer.
3
📝 Metadata Enrichment & PII Safety
EnvContext, ProcessMetrics, type-enforced sanitization
O que e: 3 camadas: EnvContext (memoized, platform/runtime), ProcessMetrics (RSS/heap/CPU delta), Agent identification (AsyncLocalStorage). PII: strings requerem cast para marker type (never). Tool input truncado a 4KB.
Por que aprender: Type-enforced PII sanitization e um pattern unico que forca code review para cada string logada.
Conceitos-chave: AnalyticsMetadata_I_VERIFIED_THIS_IS_NOT_CODE_OR_FILEPATHS (never type), CPU delta calculation, AsyncLocalStorage for agents, 4KB tool input cap.
4
🎯 GrowthBook Feature Flags
Remote eval, 3-level override, exposure dedup
O que e: Remote eval: servidor computa valores. 3 niveis de override: env var (CLAUDE_INTERNAL_FC_OVERRIDES), config (/config Gates), remote eval (api.anthropic.com). Cache em disco. SDK workaround: value -> defaultValue transform.
Por que aprender: Feature flags controlam sampling, A/B tests, kill-switches e configs dinamicas sem deploy.
Conceitos-chave: Remote eval mode, 3-level priority, empty features safety guard, exposure dedup Set, disk cache fallback.
5
🚫 Kill-Switch & Disable Conditions
tengu_frond_boric, fail-open, Bedrock/Vertex disable
O que e: Kill-switch via GrowthBook config (nome obfuscado: tengu_frond_boric). Fail-open: config ausente -> sinks habilitados. Analytics desabilitado para Bedrock/Vertex/Foundry e test environments. Feedback surveys nao afetadas.
Por que aprender: Fail-open e uma decisao de design critica: nunca silenciar dados inesperadamente.
Conceitos-chave: Obfuscated config key, fail-open design, third-party provider disable, isTelemetryDisabled(), feedback surveys separate.
6
💎 Event Sampling & Key Takeaways
Inverse probability weighting, graceful shutdown
O que e: Per-event sampling via GrowthBook config. sample_rate appended ao metadata para inverse-probability weighting. shutdownDatadog() antes de process.exit() para flush do batch final.
Por que aprender: Resume os patterns de design: zero-dep API, type-enforced PII, fail-open, swarm-aware attribution.
Conceitos-chave: Probabilistic sampling, inverse-probability weighting, graceful shutdown flush, zero-dep public API, swarm-aware from day one.

Migrations

Sistema de migracoes idempotentes: settings promotions, model alias upgrades, one-shot resets e version gate.

1
🏗️ runMigrations() & Version Gate
CURRENT_MIGRATION_VERSION=11, preAction hook
O que e: runMigrations() em main.tsx no preAction hook. migrationVersion em ~/.claude.json: quando igual a CURRENT_MIGRATION_VERSION, todo o bloco sync e skipped. Async migration (migrateChangelogFromConfig) roda fire-and-forget.
Por que aprender: O version gate evita 11 redundantes saveGlobalConfig a cada startup quando nada mudou.
Conceitos-chave: migrationVersion gate, CURRENT_MIGRATION_VERSION=11, bump rule for new migrations, async separation.
2
🔄 5 Migration Types
Settings, model alias, config rename, reset, async file
O que e: Type 1: Settings promotions (GlobalConfig -> settings.json). Type 2: Model alias upgrades (stale model strings). Type 3: Config key renames. Type 4: One-shot resets (re-surface dialogs). Type 5: Async file migrations.
Por que aprender: Classificar migracoes por tipo ajuda a escolher o pattern correto de idempotencia.
Conceitos-chave: 5 types, settings promotion, model alias upgrade, config key rename, one-shot reset, async file migration.
3
🔒 Two Idempotency Patterns
Completion flag vs self-idempotent data check
O que e: Pattern A: Completion flag em GlobalConfig (boolean/timestamp checked no topo). Pattern B: Self-idempotent (verifica valor atual, se ja migrado retorna early). Leitura e escrita na mesma source (userSettings) garante idempotencia sem flag.
Por que aprender: Escolher o pattern correto determina se a migracao e robusta ou pode ter efeitos duplicados.
Conceitos-chave: Completion flag pattern (GlobalConfig boolean), self-idempotent pattern (data speaks), same-source read/write discipline.
4
🛡️ Settings Layer Discipline
Only touch userSettings, never merged settings
O que e: Migracoes de modelo APENAS tocam userSettings. Ler merged settings promoveria um pin de projeto para o default global. Legacy strings em project/local/policy sao remapeadas em runtime por parseUserSpecifiedModel.
Por que aprender: Violar esta disciplina causaria bugs sutis onde preferencias de projeto se tornam globais.
Conceitos-chave: userSettings-only writes, never read merged settings, runtime remap via parseUserSpecifiedModel, project scope containment.
5
📋 Migration Catalogue (11 sync + 1 async)
Model migrations, MCP approval, config renames
O que e: 11 sync: autoUpdates, bypassPermissions, MCP servers, resetProToOpus, sonnet1m->4.5, legacyOpus, sonnet4.5->4.6, opus->opus1m, replBridge rename, autoModeOptIn reset, fennec (ant-only). 1 async: changelog file migration.
Por que aprender: O catalogo mostra a variedade de situacoes que requerem migracoes e os patterns usados em cada uma.
Conceitos-chave: Model migrations (sonnet, opus, fennec), settings promotions (autoUpdates, bypass), config renames (replBridge), analytics instrumentation.
6
💎 Adding a New Migration - Recipe
7-step checklist, try/catch requirement
O que e: 7 passos: criar arquivo, escolher pattern de idempotencia, apenas userSettings/localSettings, logEvent(), try/catch (nunca throw), importar em main.tsx, bumpar CURRENT_MIGRATION_VERSION. Migracoes nunca devem quebrar startup.
Por que aprender: Seguir o recipe garante que novas migracoes sao consistentes com o sistema existente.
Conceitos-chave: 7-step recipe, must never throw, logEvent for observability, bump version for existing users, @[MODEL LAUNCH] comment.

4
🔧 Command, Skill & Hook Loading

Por que aprender: Namespacing previne colisoes entre plugins. Hot reload permite habilitar/desabilitar sem restart.
Conceitos-chave: Namespaced commands/hooks/MCP, 5 substitution variables, sensitive values never in model content, hot reload via settings watcher.
5
🔒 Security, Policy & Scopes
Policy blocking, delisting, 4 install scopes
O que e: Policy blocking via managed-settings.json (enforced em install/enable/UI). Delisting com auto-uninstall quando forceRemoveDeletedPlugins=true. 4 scopes: user, project, local, managed. Reverse dependent detection no uninstall.
Por que aprender: Enterprise control sobre plugins e essencial para ambientes regulados.
Conceitos-chave: isPluginBlockedByPolicy() at 3 chokepoints, forceRemoveDeletedPlugins, 4 installation scopes, reverse dependent warning.
6
🔄 Background Autoupdate
Non-blocking, non-in-place, restart prompt
O que e: Autoupdate roda silenciosamente no background sem bloquear REPL. Updates sao non-in-place: nova versao cacheada, sessao atual usa versao antiga. REPL notificado via onPluginsAutoUpdated() com pendingNotification para race condition handling.
Por que aprender: O design non-in-place garante que sessoes nunca sao interrompidas por updates de plugin.
Conceitos-chave: Background non-blocking, non-in-place update, pendingNotification for race condition, official default autoUpdate=true.

Hooks System

27 hook events, 5 command types, exit code semantics, 6 config sources e HTTP/agent hook security.

1
📋 27 Hook Events & Exit Codes
Lifecycle, tool, agent, compaction, filesystem
O que e: 27 eventos em 7 categorias: Lifecycle (5), Tool (3), Agent (2), Compaction (2), Permission (5), Collaborative (3), Filesystem (4), MCP (2). Exit code 2 = block + tell model. Exit code 0 = success. Outros = user-visible only.
Por que aprender: Exit code 2 como "block and tell the model" e a convencao mais importante do hook protocol.
Conceitos-chave: 27 events in HOOK_EVENTS const, exit code 2 = universal block, StopFailure ignores all codes, per-event exit semantics.
2
🔧 5 Hook Command Types
command, prompt, agent, http, function
O que e: command (shell subprocess), prompt (single LLM call, JSON schema output), agent (multi-turn sub-agent ate 50 turns), http (POST to URL), function (TypeScript callback, session-only). Hierarquia de poder vs simplicidade.
Por que aprender: Cada tipo tem tradeoffs diferentes de latencia, poder e complexidade.
Conceitos-chave: 5 types hierarchy, prompt hooks don't trigger UserPromptSubmit, agent hooks use StructuredOutput tool, function hooks session-scoped only.
3
📚 6 Config Sources & Policy
Merge all, allowManagedHooksOnly, disableAllHooks
O que e: 6 fontes: user > project > local > policy > plugin > session. Todas rodam quando matching. allowManagedHooksOnly: apenas managed hooks. disableAllHooks em policy: zero hooks (unica forma de desabilitar managed hooks).
Por que aprender: Policy hooks sao o mecanismo de enforcement enterprise mais poderoso.
Conceitos-chave: 6 sources all run, allowManagedHooksOnly suppresses non-managed, disableAllHooks kills everything, non-managed cannot disable managed.
4
🔒 HTTP Hooks Security Model
