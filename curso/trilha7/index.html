Buddy Companion

Sistema Tamagotchi virtual com criatura ASCII deterministica por userId, species/eyes/hat/rarity/stats, animacoes idle e speech bubbles.

1
🎲 Deterministic Creature Generation
Mulberry32 PRNG, bones/soul split, rarity weights
O que e: Criaturas geram deterministicamente via Mulberry32 PRNG seeded por hash(userId + 'friend-2026-401'). Bones (species, eye, hat, rarity, stats) regeneram do hash; Soul (name, personality) persiste em config.
Por que aprender: O split bones/soul e tamper-proof (rarity nao pode ser editada no config) e forward-compatible.
Conceitos-chave: Mulberry32 (6 linhas), FNV-1a fallback, RARITY_WEIGHTS (60/25/10/4/1), rollStats peak/dump, rollCache.
2
🎨 ASCII Sprite Engine & Hats
3 frames, {E} template, hat slot, narrow fallback
O que e: 18 species com 3 frames de animacao cada. {E} placeholder substituido por eye char. Top row = hat slot (blank se sem hat). Abaixo de 100 colunas, colapsa para face inline de uma linha.
Por que aprender: O sprite engine demonstra rendering ASCII com template substitution e responsive layout em terminal.
Conceitos-chave: renderSprite(), IDLE_SEQUENCE com blink (-1), PET_HEARTS 5 frames, MIN_COLS_FOR_FULL_SPRITE=100.
3
💬 System Prompt & Speech Bubbles
companion_intro attachment, Claude steps aside
O que e: Companion injeta via companion_intro attachment. Claude recebe instrucao de responder em UMA linha quando usuario fala com companion por nome. Speech bubbles em dois paths: inline (scrollback) e floating (fullscreen).
Por que aprender: O prompt engineering para separar entidades (Claude vs companion) e um pattern avancado de UX conversacional.
Conceitos-chave: companionIntroText(), getCompanionIntroAttachment() dedup, BUBBLE_SHOW=20 ticks, FADE_WINDOW=6.
4
🎉 Launch Window & Species Obfuscation
April 1-7 local time, String.fromCharCode encoding
O que e: Teaser window April 1-7 2026 em hora local (nao UTC) para rolling wave de buzz. Todos 18 species codificados via String.fromCharCode para evitar canary de codename de modelo no build output.
Por que aprender: Local time vs UTC e uma decisao deliberada de marketing no codigo. A obfuscacao de species demonstra build pipeline awareness.
Conceitos-chave: isBuddyTeaserWindow() local date, isBuddyLive(), RainbowText notification, as casts erased pre-bundle.
5
📊 Stats, Shiny & Full Data Flow
5 stats RPG-style, 1% shiny chance, hatch-to-bubble
O que e: 5 stats (DEBUGGING, PATIENCE, CHAOS, WISDOM, SNARK) com peak/dump pattern. Legendary peak pode atingir 130 antes do cap 100. 1% shiny chance. Flow completo: /buddy -> bones roll -> soul gen -> intro inject -> sprite render -> bubble.
Por que aprender: O data flow end-to-end conecta PRNG, AI generation, system prompt e React rendering.
Conceitos-chave: RARITY_FLOOR scaling, inspirationSeed for AI soul gen, companionMuted config, 500ms TICK_MS.
6
❤️ Pet Hearts & Animation System
Excited vs idle mode, CompanionFloatingBubble
O que e: /buddy pet mostra 5 frames de coracoes flutuantes (2.5s). Excited mode cicla todos frames rapidamente. Idle mode segue IDLE_SEQUENCE ponderada com blinks ocasionais. Fullscreen usa CompanionFloatingBubble separado.
Por que aprender: O sistema de animacao demonstra state machine simples para UX delightful em terminal.
Conceitos-chave: PET_BURST_MS=2500, petStartTick sync-during-render, CompanionSprite vs CompanionFloatingBubble, NARROW_QUIP_CAP=24.

Ultraplan

Modo de planejamento remoto que spawna sessao CCR com Opus por 30 minutos, com keyword detection, polling engine e plan delivery dual-path.

1
🔍 Keyword Trigger & Smart Disambiguation
Filtered contexts: backticks, quotes, paths, questions
O que e: "ultraplan" em texto freeform ativa o modo, exceto dentro de delimitadores (backticks, quotes, tags), contextos de path (/\\-), seguido de ? (perguntas) ou .ext (extensoes). /ultraplan vai direto ao command handler.
Por que aprender: O keyword scanner com filtering contextual demonstra NLP pragmatico sem ML.
Conceitos-chave: Paired delimiter exclusion, path/identifier context, question mark filter, file extension filter.
2
🚀 Launch Sequence & CCR Session
teleportToRemote(), git bundle upload, detached pattern
O que e: Pattern "detached" retorna mensagem ao usuario imediatamente. Async work em void closure. ultraplanLaunching flag setado sincronamente antes de qualquer async para fechar race window de double-launch.
Por que aprender: O detached pattern com synchronous flag e aplicavel a qualquer operacao async que precisa de feedback imediato.
Conceitos-chave: teleportToRemote(), git bundle upload, ultraplanLaunching flag, void closure pattern.
3
📊 Polling Engine & Phase State Machine
Cursor-based pagination, ExitPlanModeScanner
O que e: Polling a cada 3s com ate 50 paginas por tick. ExitPlanModeScanner classifica: approved/teleport/rejected/pending/terminated. Tolera ate 5 falhas consecutivas de rede. State machine: running -> needs_input -> plan_ready -> approved/teleport.
Por que aprender: O polling engine com tolerance a falhas e cursor-based pagination e um pattern robusto para integracao com APIs remotas.
Conceitos-chave: pollForApprovedExitPlanMode(), 30-min window, cursor-based pagination, 5 consecutive failure tolerance.
4
📦 Plan Delivery: Remote vs Teleport
Path A approve in browser, Path B back to terminal
O que e: Path A: usuario aprova no browser CCR, CLI local marca task completed. Path B: usuario clica "back to terminal", browser envia sentinel __ULTRAPLAN_TELEPORT_LOCAL__, CLI mostra UltraplanChoiceDialog.
Por que aprender: O dual delivery path demonstra design de UX que respeita a escolha do usuario entre browser e terminal.
Conceitos-chave: is_error=true tool_result sentinel, UltraplanChoiceDialog, --resume via sidecar metadata files.
5
⚙️ Model Selection & Session Survival
GrowthBook model flag, --resume metadata persistence
O que e: Modelo selecionado via GrowthBook feature flag (nao hardcoded). Sessoes sobrevivem --resume via sidecar metadata files que persistem taskId, sessionId e remoteTaskType.
Por que aprender: Session survival cross-restart com sidecar files e um pattern robusto para long-running operations.
Conceitos-chave: GrowthBook model selection, sidecar metadata, plan-mode only scope, 30-min execution window.
6
🔄 Four Phases End-to-End
Trigger -> Launch -> Poll -> Deliver
O que e: Fase 1: Trigger Detection (keyword ou /ultraplan). Fase 2: CCR Session Launch (teleportToRemote). Fase 3: Long-Poll (pollForApprovedExitPlanMode, 3s interval). Fase 4: Plan Delivery (UltraplanChoiceDialog ou remote execute).
Por que aprender: A visao end-to-end conecta todas as pecas e revela como offloading de planning funciona na pratica.
Conceitos-chave: Offloading to Opus, local terminal free during planning, browser-to-terminal handoff, notification enqueue.

Entrypoints & SDK

Como Claude Code e invocado: CLI, MCP server, headless, bridge, daemon. Agent SDK com 26 lifecycle hooks e control protocol.

1
⚡ cli.tsx: Bootstrap Dispatcher & Fast Paths
8+ fast paths, zero-import --version, feature() DCE
O que e: cli.tsx pattern-matches process.argv e fast-paths 8+ comandos antes de carregar o import graph de 200+ modulos. --version imprime MACRO.VERSION inline sem nenhum import. feature() gates sao dead-code elimination em build time.
Por que aprender: O bootstrap dispatcher e o exemplo definitivo de lazy loading para startup performance.
Conceitos-chave: Zero-import --version, --daemon-worker, remote-control fast path, feature() Bun DCE gate, main.tsx fallthrough.
2
🔧 init.ts: Shared Initialization
13-step memoized init, consent-gated telemetry
O que e: init() memoizado com 13 steps: enableConfigs, CA certs, graceful shutdown, OpenTelemetry, OAuth, JetBrains detection, mTLS, preconnect API, upstream proxy, LSP cleanup, scratchpad. Telemetry so apos trust dialog.
Por que aprender: A separacao consent-independent vs consent-gated e um padrao de compliance essencial.
Conceitos-chave: Memoized init(), preconnectAnthropicApi() parallel, initializeTelemetryAfterTrust(), 13 ordered steps.
3
🔌 MCP Server Mode & SDK Contract
--mcp stdio, agentSdkTypes.ts stub pattern
O que e: --mcp expoe todos os tools do Claude Code via MCP sobre stdio para outros agentes/editores. agentSdkTypes.ts exporta API publica com funcoes stub (throw 'not implemented') - implementacoes injetadas pelo transport layer em runtime.
Por que aprender: O pattern de stub functions com runtime injection e a simetria MCP (produz e consome) sao arquiteturalmente elegantes.
Conceitos-chave: ListTools/CallTool handlers, zodToJsonSchema, thinkingConfig disabled, query() V1 stable, unstable_v2 session API.
4
📡 Control Protocol & 26 Lifecycle Hooks
SDK builders vs consumers, HOOK_EVENTS array
O que e: Control protocol JSON sobre stdio com subtypes: initialize, interrupt, can_use_tool, set_permission_mode, set_model etc. 26 hook events cobrem tool execution, permissions, session/turn lifecycle, agent/swarm, config changes, worktree.
Por que aprender: Os 26 hooks sao a API de extensibilidade mais completa do Claude Code, essencial para integracao enterprise.
Conceitos-chave: SDKControlRequest subtypes (8), BaseHookInput, agent_id vs agent_type distinction, controlSchemas.ts.
5
🖥️ Daemon & Bridge Primitives
watchScheduledTasks, connectRemoteControl @internal
O que e: @internal exports: watchScheduledTasks() (async generator de fire/missed events), connectRemoteControl() (two-way bridge WebSocket). Daemon mode: WS no parent sobrevive crash do agent child.
Por que aprender: A diferenca arquitetural daemon vs enableRemoteControl e fundamental para reliability em producao.
Conceitos-chave: ScheduledTasksHandle async generator, RemoteControlHandle write/inboundPrompts, daemon respawn resilience.
6
🛡️ Sandbox Types & Invocation Modes
SandboxSettings SSoT, main.tsx Commander modes
O que e: sandboxTypes.ts e single source of truth para config de isolamento. main.tsx oferece -p (headless), --mcp, --resume, --continue, --sdk-transport=process. Cada modo tem startup e teardown especificos.
Por que aprender: Entender os invocation modes e essencial para integracao do Claude Code em diferentes ambientes.
Conceitos-chave: SandboxSettings (network + filesystem), -p print mode, --dangerously-skip-permissions, --allowedTools comma list.

Kairos Always-On

Modo agente persistente com tick loop, SleepTool cost-aware, daily-log memory, BriefTool entitlement/activation split e AutoDream consolidation.

1
🏷️ Feature Flag Family & kairosActive Pivot
7 independent flags, positive ternaries for DCE
O que e: KAIROS e uma familia de 7 flags (KAIROS, KAIROS_BRIEF, KAIROS_PUSH_NOTIFICATION, KAIROS_GITHUB_WEBHOOKS, KAIROS_CHANNELS, PROACTIVE, AGENT_TRIGGERS). kairosActive em bootstrap/state.ts pivota memory, BriefTool, fast mode, bridge worker type.
Por que aprender: O design de flags independentes com positive ternaries para DCE e um padrao de feature management exemplar.
Conceitos-chave: feature('KAIROS') ant-only, AGENT_TRIGGERS external-shippable, kairosActive cascading effects, positive ternary rule.
2
⏰ Tick Loop & SleepTool
tengu_tick heartbeat, cost-aware idling, 5-min cache
O que e: Modelo recebe tengu_tick XML periodicamente ("you're awake, what now?"). SleepTool permite idle sem gastar API calls. System prompt explicitamente informa sobre 5-min prompt cache expiry para decisoes cost-aware.
Por que aprender: Tratar o modelo como participante cost-aware via system prompt e uma abordagem transparente e eficiente.
Conceitos-chave: tengu_tick batching, SleepTool vs Bash(sleep), QueuePriority (now/next/later) for wake, sleep_progress ephemeral.
3
🛠️ KAIROS Tool Suite
Sleep, Brief, SendFile, PushNotification, SubscribePR, Cron
O que e: 7+ tools condicionais: SleepTool (yield CPU), BriefTool/SendUserMessage (output primario), SendUserFileTool, PushNotificationTool, SubscribePRTool (GitHub webhooks), CronCreate/Delete/List.
Por que aprender: Cada tool tem logica de enable complexa separando entitlement de activation.
Conceitos-chave: isBriefEntitled() vs isBriefEnabled(), DCE gotcha (top-level feature() guard), conditional loading in tools.ts.
4
🧠 AutoDream Memory Consolidation
Forked sub-agent, 4-phase prompt, file-mtime lock
O que e: AutoDream dispara apos 24h e 5 sessoes como sub-agente forked. 4 fases: Orient (ler MEMORY.md), Gather (daily logs), Consolidate (merge), Prune (index update). Read-only Bash constraint. File-mtime lock com rollback.
Por que aprender: AutoDream e o primeiro exemplo de "agente mantendo agente" - consolidacao autonoma de conhecimento.
Conceitos-chave: cheapest-gate-first chain, rollbackConsolidationLock(), DreamTask UI visibility, 10-min scan throttle.
5
📝 Daily-Log Memory & Session Continuity
Append-only logs, --session-id, perpetual bridge
O que e: KAIROS usa daily log files (logs/YYYY/MM/DD.md) em vez de MEMORY.md. Append-only durante trabalho. --session-id e --continue permitem retomar sessao cross-restart via bridge-pointer.json.
Por que aprender: O modelo de memoria append-only e sessao persistente sao fundamentais para agentes de longa duracao.
Conceitos-chave: getAutoMemDailyLogPath(), perpetual flag falls back to env-based bridge, TEAMMEM sync disabled.
6
🚨 GrowthBook Kill-Switches & Design Principles
5-min refresh, env var overrides, 5 architectural principles
O que e: Cada subsistema tem kill-switch GrowthBook com 5-min refresh. Env vars locais (CLAUDE_CODE_DISABLE_CRON) override fleet gates. 5 principios: positive ternaries, independent shippability, cheapest gate first, operator kill switches, cost budgeting no system prompt.
Por que aprender: Os 5 principios sao aplicaveis a qualquer sistema com feature flags em escala.
Conceitos-chave: tengu_kairos_cron kill switch, tengu_frond_boric per-sink analytics, isKilled polled every tick, CACHED_WITH_REFRESH pattern.

Cost Analytics

Dual pipeline (1P OpenTelemetry + Datadog), cost tracking por sessao, PII segregation com phantom types, sampling dinamico e disk-backed retry.

1
💰 Cost Tracking & Three Destinations
addToTotalSessionCost(), OTel counters, analytics log
O que e: Cada API response distribui custo para 3 destinos: in-memory accumulators, OTel counters (BigQuery) e analytics event log. Custo em microdollars para payloads inteiros. Session costs persistem em project config JSON.
Por que aprender: Fan-out de metricas para multiplos sinks em uma unica call sincrona e um padrao de observabilidade essencial.
Conceitos-chave: addToTotalSessionCost() single entry point, cost_usd_micros, advisor sub-usage recursive, saveCurrentSessionCosts().
2
📨 Analytics Sink & Queue Pattern
Zero-dependency module, queueMicrotask drain
O que e: analytics/index.ts tem zero dependencias para evitar import cycles. Events enfileiram antes do sink conectar, drenados via queueMicrotask quando attachAnalyticsSink() e chamado. Sink roteia para Datadog (stripped) e 1P (full payload).
Por que aprender: O queue-then-drain pattern com zero dependencias e aplicavel a qualquer sistema de telemetria.
Conceitos-chave: logEvent/logEventAsync, attachAnalyticsSink() idempotent, shouldSampleEvent() fail-open, _PROTO_ key stripping.
3
📊 Datadog Pipeline & User Buckets
40-event allowlist, SHA-256 30-bucket privacy, 15s batch
O que e: ~40 events permitidos. SHA-256 userId -> 30 buckets para unique user estimation sem tracking individual. 15s flush ou 100 entries. Model/tool names normalizados. First-party API only (nao Bedrock/Vertex/Foundry).
Por que aprender: Privacy-preserving cardinality estimation via bucketing e um pattern elegante para dashboards.
Conceitos-chave: DATADOG_ALLOWED_EVENTS, getUserBucket() SHA-256 mod 30, ddtags for queryable dims, shutdownDatadog() graceful flush.
4
🛡️ 1P Pipeline & Disk-Backed Retry
OTel Logs SDK, quadratic backoff, JSONL append
O que e: Pipeline 1P usa OTel Logs SDK com BatchLogRecordProcessor. Falhas append a JSONL no disco, retry com quadratic backoff (500ms -> 2s -> 4.5s, cap 30s). Max 8 attempts. 1P LoggerProvider NUNCA registrado globalmente (isolado de customer OTLP).
Por que aprender: Disk-backed retry com quadratic backoff e provider isolation sao patterns de resiliencia de producao.
Conceitos-chave: appendEventsToFile() atomic JSONL, retryPreviousBatches() on startup, base * attempts^2, provider isolation.
5
🔐 PII Segregation & Phantom Types
never-typed markers, _PROTO_ key convention
O que e: Tipos never (AnalyticsMetadata_I_VERIFIED_THIS_IS_NOT_CODE_OR_FILEPATHS) forcam explicit as cast como paper trail para code review. _PROTO_ keys hoisted para BigQuery privilegiado, stripped antes de Datadog.
Por que aprender: Phantom types como mecanismo de code review e _PROTO_ routing sao patterns de PII hygiene em escala.
Conceitos-chave: never bottom type forcing explicit cast, stripProtoFields(), _PROTO_skill_name hoisting, compile-time vs runtime guards.
6
⚙️ GrowthBook Config & Customer OTLP
3 GrowthBook dimensions, BigQuery exporter, OTEL opt-in
O que e: GrowthBook controla 3 dimensoes: enablement (tengu_log_datadog_events), sampling (tengu_event_sampling_config) e kill-switch (tengu_frond_boric). Customer OTLP via CLAUDE_CODE_ENABLE_TELEMETRY. BigQueryMetricsExporter a cada 5 min.
Por que aprender: Live pipeline reinicializacao e customer vs internal telemetry isolation demonstram operabilidade madura.
Conceitos-chave: reinitialize1PEventLoggingIfConfigChanged(), isAnalyticsDisabled() 4 conditions, auth-aware event dispatch degradation.

Desktop App

1
🔗 Desktop Handoff & Deep Links
6-state machine, claude:// protocol, session flush
O que e: /desktop (alias /app) implementa state machine de 6 estados: checking -> prompt_download -> flushing -> opening -> success/error. Deep link: claude://resume?session=uuid&cwd=path. Session flush antes de abrir.
Por que aprender: O handoff cross-application com session persistence demonstra integracao desktop madura.
Conceitos-chave: MIN_DESKTOP_VERSION='1.1.2396', flushSessionStorage(), AppleScript for dev mode, platform-specific install detection.
2
💻 IDE Integration & Lockfiles
VS Code/JetBrains detection, ~/.claude/ide/*.lock
O que e: 18+ IDEs suportados (3 VS Code family + 15 JetBrains). Plugins escrevem lockfiles em ~/.claude/ide/port.lock com workspace, PID, transport. TERM_PROGRAM shortcut evita filesystem I/O. WSL path conversion.
Por que aprender: Lockfile-based service discovery e um pattern leve e robusto para integracao local.
Conceitos-chave: LockfileJsonContent, sort by mtime (newest first), isProcessRunning(pid) stale check, WindowsToWSLConverter.
3
🌐 Chrome Native Messaging
Host manifest, 7+ Chromium browsers, activation gates
O que e: Claude Code atua como native messaging host para extensoes Chrome. Manifest auto-instalado com wrapper script (manifests nao suportam args). 7+ browsers: Chrome, Chromium, Brave, Edge, Opera, Vivaldi, Arc.
Por que aprender: Chrome native messaging e um canal de integracao poderoso e pouco explorado.
Conceitos-chave: com.anthropic.claude_code_browser_extension, --chrome-native-host fast path, shouldEnableClaudeInChrome() 5-step check.
4
🖱️ Computer Use MCP

📦 Native Installer & MCP Import
VERSION_RETENTION_COUNT=2, MCP server cherry-pick
