Boot Sequence

O pipeline de inicializacao multi-fase que executa quando o usuario digita claude no terminal.

1
🚀 Three-Layer Boot Architecture
CLI Entrypoint, Main Function, Setup + REPL
O que e: A arquitetura de boot em tres camadas: Layer 1 (cli.tsx) com fast paths e despacho de argumentos, Layer 2 (main.tsx) com parsing Commander, migracoes e validacao de permissoes, e Layer 3 (setup.ts + replLauncher.tsx) com wiring de sessao e renderizacao React/Ink.
Por que aprender: Entender a sequencia de boot e essencial para diagnosticar problemas de inicializacao, otimizar tempo de startup e compreender como o Claude Code prepara o ambiente completo antes da primeira interacao.
Conceitos-chave: Zero-cost fast paths, Commander parsing, session wiring, React-based terminal UI, parallelization of MDM/keychain reads durante module evaluation.
2
⚡ Key Startup Optimizations
Parallelization e latencia critica
O que e: MDM policy reads e keychain reads levam 20-65ms e sao lancados durante module evaluation para executar concorrentemente com a cadeia de imports, prevenindo leituras sequenciais de adicionar latencia ao caminho critico.
Por que aprender: Compreender otimizacoes de startup permite replicar padroes de paralelizacao em suas proprias aplicacoes CLI e entender trade-offs de performance.
Conceitos-chave: Concurrent subprocess investment, startMdmRawRead(), startKeychainPrefetch(), preconnectAnthropicApi(), startDeferredPrefetches().
3
🔧 Setup Phase Sequence
Ordering critico e analytics
O que e: A sequencia de setup impoe ordem critica: setCwd() executa antes de captureHooksConfigSnapshot() para garantir hooks de projeto corretos. O beacon tengu_started dispara imediatamente apos o sink de analytics conectar.
Por que aprender: Ordering de inicializacao e uma classe comum de bugs. Entender por que certas operacoes devem preceder outras previne condicoes de corrida sutis.
Conceitos-chave: setCwd() antes de captureHooksConfigSnapshot(), tengu_started beacon, health monitoring signals, deterministic setup ordering.
4
🏎️ Bare Mode Optimization
Flag --bare para cenarios scripted/SDK
O que e: O flag --bare remove UDS messaging, teammate snapshots, inicializacao de memoria de sessao, plugin hooks e todos os deferred prefetches para cenarios scripted/SDK onde sensibilidade a latencia e maxima.
Por que aprender: Bare mode e o fast path para integracao CI/CD e SDK. Saber o que e removido ajuda a debugar comportamentos ausentes em modo headless.
Conceitos-chave: isBareMode() flag, stripped features list, latency-sensitive paths, --print mode, React never imported.
5
🌍 Global State Management
bootstrap/state.ts e latch fields
O que e: bootstrap/state.ts mantem estado com escopo de sessao incluindo identidade, rastreamento de uso, flags de modo e telemetria. "Latch fields" como afkModeHeaderLatched manteem headers de API estaveis para prevenir cache busts de prompt.
Por que aprender: Entender o estado global vs estado React e fundamental para a arquitetura de duas camadas do Claude Code.
Conceitos-chave: Session-scoped state, latch fields, prompt cache stability, identity tracking, telemetry, two-layer state model.
6
⏱️ First Keystroke to First Token
Timeline completo de cold start
O que e: O timeline de t=0ms (CLI entry) ate t+50ms (first token): MDM/keychain reads em t=1ms, module eval em t=136ms, TCP warm-up em t=160ms, primeiro render em t=190ms, deferred prefetches em t=191ms.
Por que aprender: Mapear a timeline de startup ajuda a identificar gargalos e otimizar o tempo percebido pelo usuario.
Conceitos-chave: Cold start timeline, parallel work categories, TCP warm-up, deferred prefetches, time-to-first-render.

Query Engine & LLM API

As quatro camadas entre a mensagem do usuario e a resposta no terminal: QueryEngine, queryLoop, callModel e stop hooks.

1
📨 QueryEngine.submitMessage()
Uma engine por conversa, estado mutavel
O que e: QueryEngine e uma classe stateful instanciada uma vez por conversa. Mantem historico de mensagens, uso de tokens, denials de permissao e abort controller. Cada submitMessage() e um "turn".
Por que aprender: E o ponto central de todo o fluxo de dados. SDK, REPL e remote Claude Code todos passam pelo mesmo query().
Conceitos-chave: Stateful class, mutableMessages, AbortController, transcript written before API call (crash resilience), async generator pattern.
2
🔄 queryLoop() - The While(true) Core
Sete razoes para continuar o loop
O que e: queryLoop() e um while(true) carregando um State tipado entre iteracoes. O campo transition.reason registra por que o loop continuou: max_output_tokens, reactive_compact, stop_hook_blocking, token_budget_continuation, etc.
Por que aprender: Entender as 7 transition reasons e as condicoes de terminacao e crucial para debugar comportamentos inesperados do modelo.
Conceitos-chave: Continue transitions (7 reasons), Terminal conditions (7 reasons), State object, turnCount, autoCompactTracking.
3
🌐 Streaming & API Layer
SSE stream reconstruction e StreamingToolExecutor
O que e: queryModel chama a API Anthropic via streaming e reconstrui AssistantMessages a partir de SSE events. StreamingToolExecutor inicia ferramentas enquanto o stream ainda esta aberto, reduzindo latencia.
Por que aprender: Compreender o streaming pipeline e essencial para entender latencia, tombstone messages e fallback behavior.
Conceitos-chave: SSE events (message_start, content_block_delta, message_delta), tombstone messages, StreamingToolExecutor, backfillObservableInput.
4
🛡️ withRetry() - Smart Exponential Backoff
529s, OAuth refresh, persistent mode
O que e: Cada API call passa por withRetry() com ate 10 retries. Decisoes incluem: 529 foreground-only retry, Opus fallback apos 3 erros consecutivos, OAuth 401 refresh, context overflow recalculation, persistent mode com 30-min cap.
Por que aprender: O retry layer e uma state machine sofisticada que vai muito alem de exponential backoff simples.
Conceitos-chave: FallbackTriggeredError, foreground vs background source routing, ECONNRESET/EPIPE handling, UNATTENDED_RETRY mode, jitter.
5
💾 Context Management & Autocompact
Pipeline de reducao de contexto em 5 estagios
O que e: Antes de cada API call, 5 estrategias executam em ordem fixa: applyToolResultBudget, snipCompact, microcompact, contextCollapse, e autoCompact. Cada uma reduz contexto progressivamente.
Por que aprender: Gerenciamento de contexto automatico e o que permite conversas longas sem perda de informacao critica.
Conceitos-chave: Tool result budget, HISTORY_SNIP, cached microcompact, CONTEXT_COLLAPSE, reactive compact, task_budget carry-over across compaction boundaries.
6
🔔 Stop Hooks & Token Budget
Post-turn lifecycle e auto-continue
O que e: Apos o modelo terminar, handleStopHooks() executa scripts configurados. Token budget implementa auto-continue para o SDK path com thresholds de 90% e deteccao de diminishing returns.
Por que aprender: Stop hooks e token budget controlam quando o modelo para e quando continua automaticamente.
Conceitos-chave: Blocking errors, prevent continuation, fire-and-forget side effects (memory extraction, prompt suggestions, auto-dream), COMPLETION_THRESHOLD 90%, DIMINISHING_THRESHOLD 500.

State Management

Store de 35 linhas em TypeScript, AppState com 400+ campos, integracao React via useSyncExternalStore e o padrao onChangeAppState.

1
🏗️ createStore Pattern
35 linhas, zero dependencias
O que e: Um store com tres operacoes (getState, setState, subscribe) que mapeia exatamente para useSyncExternalStore. setState recebe updater function, Object.is bail-out previne re-renders desnecessarios.
Por que aprender: Demonstra que state management sofisticado pode ser implementado sem Redux/Zustand quando a interface requerida e bem definida.
Conceitos-chave: Updater function pattern, Object.is equality, Set of listeners, onChange callback, framework-agnostic design.
2
📊 AppState Shape - 90+ Fields
DeepImmutable e seis categorias logicas
O que e: AppState com 90+ campos em 6 categorias: Session Core, UI State, Permissions, Agent & Tasks, Remote & Bridge, Subsystem State. Usa DeepImmutable com escape hatch para tipos com funcoes.
Por que aprender: AppState e a single source of truth para toda a sessao. Entender sua estrutura e essencial para qualquer modificacao no Claude Code.
Conceitos-chave: DeepImmutable, escape hatch via intersection type, SpeculationState (campo mais complexo), mutable refs para performance.
3
🔍 onChangeAppState - Side-Effect Chokepoint
Centralizacao de efeitos colaterais
O que e: Funcao unica que dispara em toda transicao de estado, fazendo diff de campos relevantes para acionar efeitos: sync de permission mode para CCR, persistencia de modelo, salvamento de globalConfig.
Por que aprender: Antes deste padrao, apenas 2 de 8+ caminhos de mutacao sincronizavam corretamente. Centralizar o diff resolve isso automaticamente.
Conceitos-chave: Diff-based side effects, externalization guard (toExternalPermissionMode), externalMetadataToAppState inverse, 6 diff blocks.
4
⚙️ React Hooks Layer
useAppState, useSetAppState, selector rules
O que e: Tres hooks: useAppState (read slice com re-render seletivo), useSetAppState (write sem subscribe), useAppStateStore (raw store). Regra critica: nunca retornar novos objetos do selector.
Por que aprender: Selector incorreto causa infinite re-render loops. Entender Object.is comparison e essencial.
Conceitos-chave: useSyncExternalStore, selector identity stability, lazy useState initializer, React Compiler memo cache.
5
📁 Selectors & Transition Helpers
Pure derivations e teammateViewHelpers
O que e: Selectors aceitam Pick<AppState, ...> para testabilidade. Transition helpers como enterTeammateView recebem setAppState como argumento, mantendo-se framework-agnosticos.
Por que aprender: O padrao de Pick para selectors e setAppState como argumento para helpers sao patterns reutilizaveis em qualquer aplicacao.
Conceitos-chave: Pick type for testability, retain/evict lifecycle (30s PANEL_GRACE_MS), circular dependency avoidance, ActiveAgentForInput discriminated union.
6
🔀 Full Data Flow
Do componente ao terminal
O que e: Fluxo completo: Component chama useSetAppState -> store.setState(updater) -> Object.is check -> onChangeAppState diffs -> listeners notificados -> useSyncExternalStore triggers re-render seletivo.
Por que aprender: Visualizar o fluxo completo conecta todas as pecas e revela por que cada decisao de design foi tomada.
Conceitos-chave: End-to-end data flow, Context vs State separation, modalContext/overlayContext as true React Context, fpsMetrics external store pattern.

System Prompt Construction

Como o Claude Code monta um system prompt de milhares de tokens a partir de secoes composiveis, cache boundaries e CLAUDE.md files.

1
🎯 Priority Resolver
Waterfall: override > coordinator > agent > custom > default
O que e: buildEffectiveSystemPrompt() implementa um waterfall estrito de prioridades. overrideSystemPrompt ignora tudo. appendSystemPrompt sempre anexa exceto com override.
Por que aprender: Saber qual prompt esta ativo em cada modo (coordinator, agent, custom) e fundamental para debugging.
Conceitos-chave: Priority waterfall, appendSystemPrompt always appends, proactive/KAIROS mode appends agent instructions to default prompt.
2
🏭 Content Factory - getSystemPrompt()
Secoes estaticas + dinamicas com boundary marker
O que e: getSystemPrompt() monta duas metades separadas por SYSTEM_PROMPT_DYNAMIC_BOUNDARY. Prefix estatico (globalmente cacheavel) e tail dinamico (por sessao).
Por que aprender: Entender a boundary e essencial para otimizar custo de tokens e cache hit rates.
Conceitos-chave: Static prefix (7 sections), dynamic tail (10 sections), CLAUDE_CODE_SIMPLE stub, Promise.all for parallel fetching.
3
📋 Dynamic Sections Registry
Memoized vs volatile, MCP instructions
O que e: Secoes dinamicas sao gerenciadas por um registry nomeado. A maioria e memoized; apenas mcp_instructions e volatile porque servidores MCP podem conectar/desconectar entre turns.
Por que aprender: O design memoized vs volatile impacta diretamente cache hit rate e custo de tokens.
Conceitos-chave: systemPromptSection (memoized), DANGEROUS_uncachedSystemPromptSection (volatile), 10 registered sections, cache keyed by name string.
4
📂 CLAUDE.md Injection
4 scopes, @include directive, frontmatter filtering
O que e: CLAUDE.md files carregados em 4 scopes: managed (/etc), user (~/.claude), project (CLAUDE.md), local (CLAUDE.local.md). Suporta @include directive e frontmatter com paths filtering.
Por que aprender: CLAUDE.md e o mecanismo primario de customizacao de comportamento. Entender prioridade e composicao e essencial.
Conceitos-chave: Four-scope hierarchy, files closer to CWD = higher priority, @include circular reference prevention, YAML frontmatter paths filtering, MEMORY.md 200 lines/25KB limit.
5
🌍 Environment Info & MCP Instructions
computeSimpleEnvInfo(), undercover mode
O que e: env_info_simple injeta CWD, git status, platform, shell, OS version, model name e knowledge cutoff. MCP instructions injeta blocos de texto de servidores conectados.
Por que aprender: O contexto ambiental permite que o modelo raciocine sobre o ambiente de execucao. Undercover mode previne vazamento de IDs de modelo interno.
Conceitos-chave: Shell detection, Windows path note, knowledge cutoff per model, mcpInstructionsDelta experimental path, undercover mode for internal builds.
6
🤖 Subagent Enhancement & Escape Hatches
DEFAULT_AGENT_PROMPT, CLAUDE_CODE_SIMPLE, proactive mode
O que e: Subagents nao usam getSystemPrompt(); partem de DEFAULT_AGENT_PROMPT com enhancement de env details. Escape hatches: CLAUDE_CODE_SIMPLE (3 linhas), proactive mode (identidade autonoma).
Por que aprender: Subagents sao o mecanismo core de multi-agent. Saber como seu prompt difere e crucial.
Conceitos-chave: enhanceSystemPromptWithEnvDetails(), computeEnvInfo() (fuller version), XML env tags, proactive autonomous-agent identity prompt.

Architecture Overview

Sintese das 49 licoes anteriores em um modelo mental completo: 6 camadas, data flow, design patterns e invariantes criticos.

1
🏗️ Six-Layer Architecture
Boot, UI Shell, State, Query Engine, Tools, Services
O que e: As 6 camadas: Boot (main.tsx, setup.ts), UI Shell (replLauncher.tsx, REPL.tsx), State (AppStateStore.ts), Query Engine (QueryEngine.ts, query.ts), Tools (tools.ts, Tool.ts), Services (API, MCP, compaction).
Por que aprender: O mapa mental de 6 camadas e o framework para localizar qualquer funcionalidade no codebase.
Conceitos-chave: Layer isolation, TypeScript/Bun/React-Ink stack, Anthropic API integration, capability registry pattern.
2
🔀 Data Flow: Prompt to Response
Sequencia completa de input a output
O que e: User enters prompt -> processUserInput() -> system prompt assembled -> transcript written to disk BEFORE API call -> query loop streams -> tools execute with permissions -> response flows through UI -> session stored as JSONL.
Por que aprender: O fluxo end-to-end conecta todas as camadas e revela decisoes criticas como transcript-before-API.
Conceitos-chave: Transcript-first reliability, JSONL session storage, async generator streaming, crash-resilient design.
3
🛡️ Permission Architecture
canUseTool() choke point, 3 modos
O que e: canUseTool() e o unico choke point para decisoes de permissao. Tres modos: default (perguntar usuario), auto (permitir safe tools), bypass (permitir tudo).
Por que aprender: Seguranca e central ao Claude Code. Entender como permissoes sao gated e essencial para customizacao segura.
Conceitos-chave: Single architectural choke point, default/auto/bypass modes, always-allow/always-deny rules, hook system integration.
4
🔄 Session Management & MCP
JSONL storage, MCPServerConnection lifecycle
O que e: Sessoes armazenadas como JSONL em ~/.claude/projects/. MCP servers inicializados antes da primeira query, passados em ToolUseContext, tools/commands/resources adicionados dinamicamente.
Por que aprender: Session management e MCP sao a infraestrutura de persistencia e extensibilidade do Claude Code.
Conceitos-chave: JSONL format, fire-and-forget writes with 100ms lazy flush, loadTranscriptFromFile(), MCPServerConnection state machine.
5
🧬 Agent Swarms & Hook System
Recursive execution e lifecycle hooks
O que e: AgentTool permite execucao recursiva com QueryEngine proprio. Em swarm mode, agentes comunicam via UDS. Hooks configurados em settings.json: PreToolUse, PostToolUse, Stop, etc.
Por que aprender: Swarms e hooks sao os mecanismos de extensibilidade mais poderosos do Claude Code.
Conceitos-chave: Recursive QueryEngine spawning, UDS messaging, 8 hook types, captureHooksConfigSnapshot() security invariant.
6
💎 Key Design Patterns & Invariants
6 patterns criticos, invariantes de seguranca
O que e: 6 patterns: async generator threading, dead code elimination via feature(), cache-warming, immutable AppState + mutable bootstrap/state, isBareMode() fast path, parallel subprocess investment.
Por que aprender: Estes patterns sao a essencia do design do Claude Code e sao aplicaveis a qualquer aplicacao TypeScript de grande escala.
Conceitos-chave: Generators all the way down, feature() dead code elimination, transcript-first reliability, hooks captured after setCwd(), tool list stable per build.
